### 한 줄 정의
> **TEXT2SQL(Text-to-SQL)**: 사용자의 **자연어 질문**을 **구조화된 SQL 질의**로 변환해 DB에서 직접 조회·응답하는 기술

### 쉬운 비유
- TEXT2SQL = “사람 말(자연어)을 **DB가 알아듣는 주문서(SQL)** 로 자동 번역하는 통역기”

### 왜 필요한가? (3줄 요약)
- 데이터 활용 수요는 늘지만, 누구나 SQL을 잘 쓰긴 어렵다(비전문가 장벽).
- 자연어 처리/LLM 발전으로 자연어를 SQL로 바꿔 **빠른 의사결정**을 지원할 수 있다.
- 다만 스키마 매칭/정확도/보안(권한) 문제가 있어 **검증·통제**가 필수다.

## 나. TEXT2SQL 활용 사례 2가지

### 사례 1) BI/경영 리포트 자동 질의 `비`

**질문 예시**
- “지난달 **지역별 매출 Top 10 제품** 보여줘”
- “이번 분기 **재구매율이 떨어진 채널**은 어디야?”
- “신규 가입 대비 **유료 전환율**이 높은 캠페인 5개 알려줘”

| 항목 | 내용 |
|:--|:--|
| **업무 상황** | 경영/현업이 수시로 KPI를 조회하나, SQL 작성 능력 부족으로 분석팀 의존도가 높음 |
| **대상 데이터** | 주문/결제/고객/상품/캠페인 등 DWH(또는 마트) 테이블, 날짜·지역·채널 차원 테이블 |
| **처리 흐름** | 자연어 질의 → (스키마 링크/용어사전) → SQL 생성 → (권한/정합성/성능) 검증 → 실행 → 결과 요약 + 표/차트 생성 |
| **산출물** | 표/차트/대시보드 위젯, “근거 SQL” 링크(감사/재현성) |
| **효과** | 셀프서비스 분석 확산, 리포트 생성 리드타임 단축, 데이터 활용(의사결정) 속도 향상 |
| **운영 포인트** | 스키마/지표 정의(메트릭 레이어) 고정, 생성 SQL 화이트리스트/쿼리 가드레일(시간·스캔량 제한) 필요 |

#### 절차/매커니즘 (2그룹 × 4개) `이정거품 / 생검실응`

**그룹 1: 질의 이해·정규화 요소 `이정거품`**

| 항목 | 설명 | 예시 |
|:--|:--|:--|
| **의(이)**도 파악 | 질문의 KPI/대상/기간/필터를 구조화 | “지난달”, “Top 10”, “지역별” |
| **정(정)**의 매핑 | KPI 정의를 메트릭 레이어로 매핑 | 매출=결제금액? 주문금액? |
| **거(거)**버넌스/권한 | 테넌트/조직/권한 기반 접근 통제 | RBAC, 데이터 마트 분리 |
| **품(품)**질 가드레일 | 모호질의 재질문, 안전한 SQL만 허용 | 금지 키워드, LIMIT 강제 |

**그룹 2: SQL 생성·검증·실행·응답 `생검실응`**

| 항목 | 설명 | 예시 |
|:--|:--|:--|
| **생**성 | SQL 초안 생성(LLM/템플릿) | SELECT/JOIN/GROUP BY |
| **검**증 | 문법·스키마·정책·성능 사전검증 | 스키마 링크 체크, 스캔량 |
| **실**행 | 실행 및 최적화(재작성/힌트) | EXPLAIN, 쿼리 리라이트 |
| **응**답 | 요약 + 시각화 + 근거 SQL 제공 | 표/차트, 대시보드 위젯 |

**SQL 예시(개념)**

```sql
-- “지난달 지역별 매출 Top 10 제품”
SELECT
  r.region_name,
  p.product_name,
  SUM(o.amount) AS sales
FROM fact_orders o
JOIN dim_region r  ON o.region_id  = r.region_id
JOIN dim_product p ON o.product_id = p.product_id
WHERE o.order_date >= DATE_TRUNC('month', CURRENT_DATE - INTERVAL '1' month)
  AND o.order_date <  DATE_TRUNC('month', CURRENT_DATE)
GROUP BY r.region_name, p.product_name
ORDER BY sales DESC
LIMIT 10;
```

### 사례 2) 고객지원/운영 모니터링 질의 `고`

**질문 예시**
- “최근 3개월 **불만 접수 TOP 5 유형**은?”
- “오늘 00시 이후 **장애 이벤트** 발생 건수랑 영향 서비스는?”
- “배포 이후 **에러율 급증**한 API가 있나?”

| 항목 | 내용 |
|:--|:--|
| **업무 상황** | 상담/운영 인력이 반복적으로 KPI·장애 지표를 조회하며, 속도/정확성이 중요 |
| **대상 데이터** | CRM 티켓, 콜로그, 장애/이벤트 로그, APM/관제 지표, 배포 이력, 서비스 카탈로그 |
| **처리 흐름** | 자연어 질의 → (용어 정규화: “불만/클레임/컴플레인”) → SQL 생성 → (권한/PII 마스킹/레이트리밋) 검증 → 실행 → 요약 답변 + 근거 데이터(링크) |
| **효과** | 상담 응답 시간 단축, 운영 이상 징후 조기 감지, 반복 리포팅 자동화(야간/교대 인수인계 포함) |
| **보안 포인트** | RBAC/ABAC로 고객정보 접근 통제, PII 컬럼 자동 마스킹(또는 뷰/정책), 쿼리 감사로그 필수 |
| **품질 포인트** | “의도-쿼리-결과” 삼자검증(검증 루프), 실행 실패/모호 질의 시 재질문(Interactive refinement) |

#### 절차/매커니즘 (2그룹 × 4개) `용문권비 / 생검실응`

**그룹 1: 운영 질의 전처리·통제 요소 `용문권비`**

| 항목 | 설명 | 예시 |
|:--|:--|:--|
| **용(용)**어 정규화 | 동의어/약어/표현을 표준 용어로 변환 | 장애=Incident, 에러율=5xx rate |
| **문(문)**맥 보강 | 시간창/서비스/환경(Prod/Stage) 등 컨텍스트 주입 | “오늘”, “배포 이후” |
| **권(권)**한·PII | 민감정보 접근 제한 및 마스킹/뷰 적용 | 전화/이메일 마스킹 |
| **비(비)**상 대응 | 경보 기준/플레이북 연계, 결과를 액션으로 연결 | 알림 발송, 티켓 생성 |

**그룹 2: SQL 생성·검증·실행·응답 `생검실응`**

| 항목 | 설명 | 예시 |
|:--|:--|:--|
| **생**성 | 관제/CRM 스키마에 맞춰 질의 생성 | 로그 집계, 윈도우 함수 |
| **검**증 | 레이트리밋/정책/쿼리 안전성 검증 | 시간 제한, 금지 조인 |
| **실**행 | 운영 DB/데이터레이크에서 실행 | 파티션 프루닝, 인덱스 |
| **응**답 | 요약 + 근거 링크 + 다음 액션 제안 | “원인 후보”, “조치 추천” |

**SQL 예시(개념)**

```sql
-- “최근 3개월 불만 접수 TOP 5 유형”
SELECT
  t.issue_type,
  COUNT(*) AS cnt
FROM crm_tickets t
WHERE t.created_at >= CURRENT_DATE - INTERVAL '3' month
  AND t.ticket_category = 'COMPLAINT'
GROUP BY t.issue_type
ORDER BY cnt DESC
LIMIT 5;
```

## 참고(가산점 포인트)

- TEXT2SQL은 “정확도”를 위해 **스키마 변경 대응**, **검증 루프(Feedback/Interactive Refinement)**, **권한 통제(ACL/RBAC)** 가 중요하다.
