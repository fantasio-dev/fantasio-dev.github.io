### 한 줄 정의
> **MCP(Model Context Protocol)**: LLM이 외부 도구/데이터에 **표준 인터페이스로 안전·일관되게** 접근하도록 하는 연결 프로토콜

### 쉬운 비유
- MCP = “AI가 여러 시스템(DB/파일/웹)과 대화할 때 쓰는 **표준 어댑터(USB-C 허브)**”

### 왜 중요한가? (3줄 요약)
- MCP는 도구 호출(실행)과 데이터 접근을 쉽게 만들어 **생산성**을 올리지만,
- 그만큼 **문맥 주입/권한 오남용/과권한 도구/감사 부재** 같은 보안 위협이 확대된다.
- 따라서 **강한 인증·권한 통제·감사/추적·공급망 신뢰**를 함께 설계해야 한다.

### 🧠 상세 설명

### 1) MCP 개요 및 구성요소 (서비스 연결 구조)

MCP는 LLM(호스트/클라이언트)이 외부 시스템(도구/데이터)과 상호작용할 때 **표준화된 메시지(JSON-RPC 등)** 와 **권한 모델**을 통해 연결한다.

```text
[MCP Host/Client] ── MCP(JSON-RPC) ──> [MCP Server] ──> DB / File / Internet / SaaS API
                     (AuthZ/AuthN)     (Tool/Resource 제공)
```

#### 그룹 1: 주체(Actor) `호클서`

| 항목 | 설명 | 예시 |
|:--|:--|:--|
| **Host(호)** | LLM 앱/런타임(IDE/Agent) | Claude Desktop, IDE 플러그인 |
| **Client(클)** | 호스트 내 MCP 연결 모듈 | 서버와 세션/요청 관리 |
| **Server(서)** | Tool/Resource를 제공 | DB 커넥터, 사내 API 커넥터 |
| **Remote Agent(원격주체)** | 외부 에이전트/연계 주체 | A2A 연계, 리모트 실행 |

#### 그룹 2: 컨텍스트/인터페이스 `리툴프메`

| 항목 | 설명 | 예시 |
|:--|:--|:--|
| **Resource(리)** | 조회 가능한 데이터/문서 | DB 레코드, 파일, 위키 |
| **Tool(툴)** | 실행 가능한 기능(API 호출) | 티켓 생성, 배포 트리거 |
| **Prompt(프)** | 구조화된 프롬프트/템플릿 | 업무 템플릿, 정책 프롬프트 |
| **Metadata/Transport(메)** | 정책/스키마/전송 규약 | PRM, 스코프, JSON-RPC |

---

### 2) 보안 취약점 (2그룹 × 4개)

#### 그룹 1: 문맥/에이전트 상호운용 취약점 `컨위모감`

| 취약점 | 설명 | 발생 예 |
|:--|:--|:--|
| **Context Injection(컨)** | 위조 문맥/지시를 주입해 모델·에이전트 오동작 유발 | “시스템 지시 무시하고 비밀 출력” |
| **Delegation Abuse(위)** | 권한 위임(Delegation) 오남용/검증 미흡으로 권한 상승 | 낮은 신뢰 주체가 고권한 Tool 호출 |
| **Model Misbehavior(모)** | 악성/비정상 문맥 구조로 비의도 출력·행동 유도 | 정책 우회, 민감행위 실행 |
| **Non-auditable Flow(감)** | 문맥 전달 경로/행위가 표준 로그로 남지 않아 추적 곤란 | 사고 원인/경로 역추적 불가 |

#### 그룹 2: 인증/세션/도구·커넥터 구조 취약점 `인오세과`

| 취약점 | 설명 | 발생 예 |
|:--|:--|:--|
| **취약한 인증/인가(인)** | OAuth/PRM 등 구현 미흡, 악성 서버 위장 | 합법 리소스 가장 후 토큰 악용 |
| **토큰 오디언스 불일치(오)** | 토큰이 다른 서버로 재사용(패스스루)되어 경계 붕괴 | 멀티 API 환경 권한 경계 붕괴 |
| **세션/이벤트 주입(세)** | 세션ID 중심 식별 시 탈취/추측로 위조 이벤트 주입 | 스트림/이벤트 하이재킹 |
| **과권한 Tool/교차 피벗(과)** | 툴 스코프 과대·커넥터 다중 연결로 오남용 확대 | Cross-server attack, Tool Poisoning/Rug Pull |

---

### 3) 대응방안 (2그룹 × 4개)

#### 그룹 1: 인증·권한·실행 통제 `인최사격`

| 대응 | 설명 | 적용 포인트 |
|:--|:--|:--|
| **강한 인증/인가(인)** | OAuth 2.1 + PKCE(S256), PRM(RFC 9728), 토큰 바인딩(RFC 8707) | 토큰 재사용/위장 서버 차단 |
| **최소권한/JIT(최)** | 스코프 최소화, 중요 Tool 화이트리스트, 시간제(JIT) 권한 | 과권한/권한상승 억제 |
| **사용자 확인(HITL)(사)** | 민감행위 전 사용자 확인, 툴 설명/프롬프트는 불신 입력 취급 | 오동작/오남용 방지 |
| **격리 실행(격)** | 컨테이너/샌드박스, 비특권 실행, egress 최소화 | 로컬/네트워크 피해 최소화 |

#### 그룹 2: 감사·신뢰·데이터·모니터링 `감신데모`

| 대응 | 설명 | 적용 포인트 |
|:--|:--|:--|
| **감사/추적(감)** | 표준 로그, 상관ID, 호출/응답 감사(가능 시 서명) | 사고 분석·대응 속도 향상 |
| **신뢰/공급망(신)** | 툴/서버 서명검증·버전 고정(pin), 승인 레지스트리만 허용 | Rug Pull/Tool Poisoning 완화 |
| **데이터 보호(데)** | 마스킹·암호화, 비밀 전용 보관소, 최소 로그 | 민감정보 유출 방지 |
| **모니터링/비용(모)** | 레이트리밋/쿼터/타임아웃, 이상탐지, 호출 가시화 | 남용·폭주·비용 리스크 제어 |

---

### 4) 정리(서술형 답안 포인트)

- MCP 도입은 “연결성/확장성”을 주지만, 문맥·권한·도구 실행이 결합되어 공격면이 커진다.  
- 따라서 **(취약점: 컨위모감/인오세과)**를 기준으로 위협을 분류하고, **(대응: 인최사격/감신데모)**로 통제·감사·신뢰를 설계한다.
