<details markdown="block">

### 프록시가 뭐야?

**프록시(Proxy)**는 우리말로 **"대리자"**라는 뜻이에요!

생각해보세요. 연예인이 직접 모든 팬미팅 요청을 받으면 너무 바쁘겠죠? 그래서 **매니저**가 대신 연락을 받고, 일정을 조율해요. 이 매니저가 바로 **프록시**예요!

```
👤 팬(Client) → 📱 매니저(Proxy) → ⭐ 연예인(RealSubject)
```

### 왜 프록시를 쓸까?

| 이유 | 예시 |
|:---|:---|
| **접근 제어** | VIP 팬만 만나게 하고 싶어요 → 매니저가 체크! |
| **지연 로딩** | 연예인이 준비될 때까지 기다려요 → 매니저가 대기 처리! |
| **로깅** | 누가 연락했는지 기록해요 → 매니저가 메모! |

### 프록시 종류

| 종류 | 한줄 설명 |
|:---|:---|
| 🎭 **가상 프록시** | 무거운 거 나중에 불러와 (이미지 로딩) |
| 🔒 **보호 프록시** | 권한 체크해! (VIP만 입장) |
| 🌐 **원격 프록시** | 멀리 있는 것도 접근해 (API 호출) |
| 🧠 **스마트 프록시** | 추가 기능 붙여 (로깅, 캐싱) |

### 암기팁 🎯

**"프록시는 대리자! 진짜 대신 일 처리해주는 친구!"**

## 기술사수준 답안

### 1. 대리자 통한 접근 및 제어, 프록시 디자인 패턴의 개념

| 구분 | 내용 |
|:---|:---|
| **개념** | 객체에 직접 접근하지 않도록 같은 인터페이스를 가진 프록시를 구성하여 요청을 대신 받아 처리하거나 전달하는 디자인 패턴 |
| **목적** | 접근 제어, 로깅, 지연 로딩, 캐싱 등 부가 기능 수행 |

---

### 2. 프록시 디자인 패턴의 구성도 및 구성요소

#### 구성도

```
┌─────────┐    request()    ┌───────────────────────────────────────┐
│ Client  │ ─────────────► │      <<interface>> Subject            │
└─────────┘                 │  ┌─────────────────────────────────┐  │
                            │  │        + request()              │  │
                            │  └─────────────────────────────────┘  │
                            └───────────────────────────────────────┘
                                            △
                                            │
                    ┌───────────────────────┴───────────────────────┐
                    │                                               │
            ┌───────────────┐                               ┌───────────────┐
            │     Proxy     │ ─────────────────────────────►│  RealSubject  │
            ├───────────────┤       realSubject              ├───────────────┤
            │ + request()   │                               │ + request()   │
            └───────────────┘                               └───────────────┘
```

- 클라이언트가 실제 객체(RealSubject) 대신 프록시를 통해 동일한 인터페이스로 요청을 전달하도록 구성한 구조

---

#### 구성요소 (패턴 구성) — (패턴) `주실대요`

| 구성 요소 | 설명 |
|:---|:---|
| **Subject (주체)** | 공통 인터페이스, RealSubject와 Proxy가 구현해야 할 계약 정의 |
| **RealSubject (실체)** | 실제 작업을 수행하는 객체, 핵심 비즈니스 로직 포함 |
| **Proxy (대리자)** | RealSubject를 대신해 클라이언트 요청 처리, 접근 제어/부가 기능 수행 |
| **Client (요청자)** | Subject 인터페이스를 통해 요청을 보내는 객체 |

---

#### 프록시 유형 — (유형) `가보원스`

| 유형 | 설명 |
|:---|:---|
| **가상 프록시 (Virtual)** | 무거운 객체를 나중에 생성 (ex. 이미지 지연 로드) |
| **보호 프록시 (Protection)** | 접근 권한 제어를 수행 (ex. 관리자/사용자 구분) |
| **원격 프록시 (Remote)** | 다른 주소공간(서버)에 있는 객체 접근 (ex. RMI, gRPC) |
| **스마트 프록시 (Smart)** | 로깅, 참조 횟수 관리 등 부가기능 포함 |

- 프록시 디자인 패턴은 Spring 외에도 다양한 프레임워크나 시스템에서 핵심적인 역할 수행

---

### 3. 프록시 디자인 패턴 활용 프레임워크

| 구분 | 프레임워크 | 설명 |
|:---|:---|:---|
| 웹 프레임워크 | Django (Python) | 미들웨어 구조에서 프록시 역할로 요청/응답 처리 |
| 네트워크/웹 | Apache Http Server | 리버스 프록시: 클라이언트 요청을 서버에 전달 |
| 마이크로서비스 | Service Mesh | 사이드카 프록시 통한 서비스간 통신 제어 |
| ORM | Hibernate | Lazy Loading을 위해 프록시 객체 생성 |
| 보안 | Squid Proxy | 접근 제어, 요청 분석, 필터링 등 수행 |
| AI/ML 플랫폼 | TensorFlow | 모델 앞단에 gRPC 프록시 통한 라우팅, 캐싱 |

- 다양한 프레임워크와 기술에서 요청 제어, 접근 제어, 지연 로딩, 트래픽 관리 등 목적으로 다양하게 활용
