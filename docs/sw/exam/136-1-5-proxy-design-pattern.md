### 📖 등장배경

```
🌐 과거: 객체에 직접 접근하면 보안, 성능, 네트워크 문제 발생

💡 문제: "무거운 객체를 바로 로딩하면 느리고, 권한 없는 사람도 접근하면 어쩌지?"

✅ 해결: 프록시 디자인 패턴으로 해결!
   → 프록시가 대신 요청을 받아서 처리
   → 접근 제어, 지연 로딩, 로깅 등 부가 기능 수행
   → 실제 객체는 필요할 때만 호출
```

### 📝 개념 정의

> **프록시 디자인 패턴**: 진짜 대신 대리자가 요청을 처리하는 패턴

| 구분 | 정의 |
|:-----|:-----|
| **프록시** | 실제 객체 대신 요청을 받는 대리 객체 |
| **Subject** | 프록시와 실제 객체가 따르는 공통 인터페이스 |

### 쉬운 비유

| 프록시 | 실제 상황 |
|:------|:---------|
| 📱 매니저 | 연예인 대신 스케줄 조율 |
| 🏠 부동산 중개사 | 집주인 대신 계약 처리 |

### 🏗️ 기술요소

#### 그룹 1: 패턴 구성요소 `주실대요`

| 항목 | 설명 | 예시 |
|:-----|:-----|:-----|
| **주**체(Subject) | 공통 인터페이스 정의 | interface Subject |
| **실**체(RealSubject) | 실제 작업 수행 객체 | 실제 이미지 로더 |
| **대**리자(Proxy) | 요청 대신 처리/전달 | 프록시 객체 |
| **요**청자(Client) | 요청을 보내는 객체 | 사용자 코드 |

#### 그룹 2: 프록시 유형 `가보원스`

| 항목 | 설명 | 예시 |
|:-----|:-----|:-----|
| **가**상 프록시 | 무거운 객체 나중에 생성 | 이미지 지연 로드 |
| **보**호 프록시 | 접근 권한 제어 수행 | 관리자/사용자 구분 |
| **원**격 프록시 | 다른 주소공간 객체 접근 | RMI, gRPC 호출 |
| **스**마트 프록시 | 로깅, 캐싱 등 부가기능 | 참조 횟수 관리 |

### ⭐ 차별점 키워드

**프록시 vs 데코레이터**: 프록시는 **접근 제어**, 데코레이터는 **기능 확장**

## 🎯 기술사 수준 설명

{: .highlight }
> **프록시 디자인 패턴**: 직접 접근 제한 필요 → 동일 인터페이스, 대리자, 접근제어 → 객체 접근 제어 구조 패턴
> - (패턴) `주실대요` (주체, 실체, 대리자, 요청자)
> - (유형) `가보원스` (가상, 보호, 원격, 스마트)
> - ⭐ **차별점**: 동일 인터페이스로 실제 객체 대신 요청 처리 (접근 제어 목적)

---

### 📝 개념 정의

| 구분 | 정의 |
|:-----|:-----|
| **개념** | 객체에 직접 접근하지 않도록 같은 인터페이스를 가진 프록시를 구성하여 요청을 대신 받아 처리하거나 전달하는 디자인 패턴 |
| **목적** | 접근 제어, 로깅, 지연 로딩, 캐싱 등 부가 기능 수행 |
| **분류** | GoF 디자인 패턴 - 구조(Structural) 패턴 |

---

### 🏗️ 구성요소

#### 그룹 1: 패턴 구성요소 `주실대요`

| 항목 | 설명 | 예시 |
|:-----|:-----|:-----|
| **주**체(Subject) | 공통 인터페이스, RealSubject와 Proxy가 구현해야 할 계약 정의 | interface Subject { request() } |
| **실**체(RealSubject) | 실제 작업을 수행하는 객체, 핵심 비즈니스 로직 포함 | ImageLoader, DBConnection |
| **대**리자(Proxy) | RealSubject를 대신해 클라이언트 요청 처리, 접근 제어/부가 기능 수행 | ImageProxy, ConnectionProxy |
| **요**청자(Client) | Subject 인터페이스를 통해 요청을 보내는 객체 | Application, Service Layer |

#### 그룹 2: 프록시 유형 `가보원스`

| 항목 | 설명 | 예시 |
|:-----|:-----|:-----|
| **가**상 프록시(Virtual) | 무거운 객체를 나중에 생성 (Lazy Loading) | 이미지 지연 로드, 문서 미리보기 |
| **보**호 프록시(Protection) | 접근 권한 제어를 수행 | 관리자/사용자 권한 구분, ACL |
| **원**격 프록시(Remote) | 다른 주소공간(서버)에 있는 객체 접근 | RMI, gRPC, REST Client |
| **스**마트 프록시(Smart) | 로깅, 참조 횟수 관리 등 부가기능 포함 | 캐싱 프록시, 로깅 프록시 |

---

### 🔄 프록시 디자인 패턴 구성도

```
┌─────────┐    request()    ┌───────────────────────────────────────┐
│ Client  │ ─────────────► │      <<interface>> Subject            │
└─────────┘                 │  ┌─────────────────────────────────┐  │
                            │  │        + request()              │  │
                            │  └─────────────────────────────────┘  │
                            └───────────────────────────────────────┘
                                            △
                                            │
                    ┌───────────────────────┴───────────────────────┐
                    │                                               │
            ┌───────────────┐                               ┌───────────────┐
            │     Proxy     │ ─────────────────────────────►│  RealSubject  │
            ├───────────────┤       realSubject              ├───────────────┤
            │ + request()   │                               │ + request()   │
            │ - preProcess  │                               │ + doWork()    │
            │ - postProcess │                               └───────────────┘
            └───────────────┘
```

- 클라이언트가 실제 객체(RealSubject) 대신 프록시를 통해 동일한 인터페이스로 요청을 전달하도록 구성한 구조

### 📚 프록시 디자인 패턴 활용 프레임워크

| 구분 | 프레임워크 | 설명 |
|:-----|:----------|:-----|
| 웹 프레임워크 | Django (Python) | 미들웨어 구조에서 프록시 역할로 요청/응답 처리 |
| 네트워크/웹 | Apache Http Server | 리버스 프록시: 클라이언트 요청을 서버에 전달 |
| 마이크로서비스 | Service Mesh | 사이드카 프록시 통한 서비스간 통신 제어 |
| ORM | Hibernate | Lazy Loading을 위해 프록시 객체 생성 |
| 보안 | Squid Proxy | 접근 제어, 요청 분석, 필터링 등 수행 |
| AI/ML 플랫폼 | TensorFlow | 모델 앞단에 gRPC 프록시 통한 라우팅, 캐싱 |

### ⭐ 차별점 키워드

| 키워드 | 설명 |
|:------|:-----|
| **동일 인터페이스** | Proxy와 RealSubject가 같은 인터페이스 구현 (투명한 대체) |
| **접근 제어 목적** | 데코레이터(기능 확장)와 달리 접근 제어가 주 목적 |
| **구조 패턴** | GoF 디자인 패턴 중 구조(Structural) 패턴 분류 |
| **Spring AOP** | 프록시 기반 AOP로 트랜잭션, 로깅, 보안 등 횡단 관심사 처리 |
