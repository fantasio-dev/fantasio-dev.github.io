### 💡 한 줄 정의

> **UML 행위 다이어그램**: 시스템의 동적 행위와 상호작용을 표현하는 UML 다이어그램 유형

### 🎭 쉬운 비유

| 다이어그램 | 비유 |
|:---------|:-----|
| **Activity Diagram** | 요리 레시피 순서도 - "이렇게 하고 → 저렇게 하고" |
| **State Diagram** | 자판기 상태 변화 - "대기 → 동전투입 → 음료선택 → 배출" |
| **Use-Case Diagram** | 영화 시놉시스 - "누가 뭘 한다" (사용자 관점) |

### ❓ 왜 필요한가? (3줄 요약)

1. **동적 행위 표현**: 구조(Structure)보다 동작(Behavior)을 시각화
2. **이해관계자 소통**: 개발자-기획자-고객 간 공통 언어 제공
3. **설계·검증 기반**: 테스트 케이스 도출 및 구현 검증 근거

### 1. UML 행위 다이어그램의 개요

{: .highlight }
> **UML 행위 다이어그램**: 동적 행위, 상호작용, 시각적 표현
> - (개념) `동상상`
> - (종류) `활상유시인`
> - (역할) `업유프`

#### 가. UML 행위 다이어그램의 개념

> **UML 행위 다이어그램**: 시스템이 "무엇을 하는가(What it does)"를 모델링하는 다이어그램으로, 동적 행위와 상호작용을 표현하는 UML 다이어그램 유형

| 구분 | 정의 |
|:-----|:-----|
| **행위 다이어그램** | 시스템의 기능과 사용자의 행위, 객체 간 상호작용 흐름을 기술 |
| **구조 다이어그램** | 시스템의 정적 구조(클래스, 컴포넌트, 배치 등)를 기술 |

#### 나. UML 다이어그램 분류

```
┌─────────────────────────────────────────────────────────────────────┐
│                        UML 다이어그램                                 │
├───────────────────────────────┬─────────────────────────────────────┤
│     구조 다이어그램              │         행위 다이어그램              │
│   (Structure Diagram)         │      (Behavior Diagram)            │
├───────────────────────────────┼─────────────────────────────────────┤
│  • 클래스 다이어그램 (Class)     │  • 활동 다이어그램 (Activity)        │
│  • 객체 다이어그램 (Object)      │  • 상태 다이어그램 (State)          │
│  • 컴포넌트 다이어그램           │  • 유스케이스 다이어그램 (Use-Case)  │
│    (Component)                │  • 시퀀스 다이어그램 (Sequence)      │
│  • 배치 다이어그램 (Deployment) │  • 인터랙션 다이어그램 (Interaction) │
│  • 패키지 다이어그램 (Package)  │                                    │
└───────────────────────────────┴─────────────────────────────────────┘
```

#### 다. 행위 다이어그램의 역할

| 역할 | 설명 |
|:-----|:-----|
| **업**무 분석 | As-Is/To-Be 업무 흐름 분석, 비즈니스 프로세스 정의 |
| **유**스케이스 실현 | 기능 요구사항을 Use Case로 정의하고 실현 |
| **프**로세스 흐름 시각화 | 제어 흐름, 상태 전이, 상호작용 등 동적 행위 표현 |

### 2. 활동 다이어그램 (Activity Diagram) 설명

{: .highlight }
> **Activity Diagram**: Workflow, 제어흐름, 병렬처리
> - (정의) `워플동적`
> - (구성요소) `시종활선전구`
> - (역할) `업유프`

#### 가. Activity Diagram의 정의

> **Activity Diagram**: 활동 처리 로직이나 조건에 따른 객체 간의 제어 흐름을 중점적으로 보여 주는 **Workflow 흐름도 기반의 동적 다이어그램**

| 역할 | 설명 |
|:-----|:-----|
| ① 업무 분석 As-Is/To-be | 현행/목표 업무 프로세스 분석 |
| ② Use Case 실현 | 유스케이스 시나리오를 활동 흐름으로 표현 |
| ③ 프로세스 흐름 시각화 | 순차/병행 프로세스 지원, 순서 표현 |

#### 나. Activity Diagram 구성도

```
        ●  ← 시작점 (Initial State)
        │
        ▼
   ┌─────────────┐
   │  Activity1   │  ← 활동 (Activity)
   └─────────────┘
        │
        ▼
       ◇  ← 조건 (특정조건에 따른 분기)
      / \       (Decision)
     ▼   ▼
┌─────────┐ ┌─────────┐
│Activity2│ │Activity3│  ← 동시경로
└─────────┘ └─────────┘    (특정 활동이 합쳐지거나,
     \       /             모이는 경우)
      ▼   ▼
   ┌─────────────┐
   │  Activity4   │
   └─────────────┘
        │
        ▼
       (◎) ← 종료점 (Final State)
```

#### 다. Activity Diagram 구성요소 `시종활선전구`

| 구성요소 | 설명 | 표기법 |
|:---------|:-----|:------|
| **시**작점 (Initial State) | 처리흐름이 시작되는 곳을 의미 | ● |
| **종**료점 (Final State) | 처리흐름이 종료되는 곳을 의미 | ◎ |
| **활**동 (Activity) | 행위나 작업(내부적으로 구조를 가지는 단위)을 무언가를 하고 있는 상태 | ▭ |
| **선**택점 (Decision) | 논리식의 결과에 따라 분기가 일어나는 곳 | ◇ |
| **전**이 (Transition) | 하나의 상태에서 다른 상태로의 제어 흐름을 보여주는데 사용, 상태에서 활동으로의, 또는 상태들 사이의 흐름 | → |
| **구**획면 (Swim Lane) | 업무조직이나 개인의 역할에 따른 처리구분 (각 활동 상태를 담당하는 역할) | 구분영역 |

### 3. 상태 다이어그램 (State Diagram) 설명

{: .highlight }
> **State Diagram**: 객체 상태 변화, 이벤트 기반, 수명주기
> - (정의) `수상이전`
> - (구성요소) `시종상전`
> - (역할) `상동구`

#### 가. State Diagram의 정의

> **State Diagram**: 시스템 행위를 명세하기 위해 대상 행위를 **상태(state)들의 집합으로 구분** 다음, 그 상태들 간의 **전이(transition) 관계와 조건을 명세**한 다이어그램

| 역할 | 설명 |
|:-----|:-----|
| ① 상태/전이 과정 시각화 | 객체의 상태가 아닌 처리 로직이나 조건에 따른 처리흐름을 순서에 따라 정의한 다이어그램 |
| ② 동작 흐름 명확화 | 객체의 상태 변화와 이벤트 기반 동작을 표현 |
| ③ 구현 테스트 근거 제공 | 상태 전이 테스트 케이스 도출 근거 |

#### 나. State Diagram 개념도

```
┌───────────────────────────────────────────────────────────────┐
│                         PC 상태 예시                           │
├───────────────────────────────────────────────────────────────┤
│                                                               │
│    ●                                                          │
│    │                                                          │
│    ▼                                                          │
│  ┌─────┐    GUI 동작     ┌──────────┐                         │
│  │ Idle │ ────────────→ │ Do/Bootup │                         │
│  └─────┘                 └──────────┘                         │
│    ▲                          │                               │
│    │                          ▼                               │
│    │                   ┌───────────────┐                      │
│    │                   │ Shutting Down │                      │
│    │                   └───────────────┘                      │
│    │ Time Out                 │                               │
│    │                          ▼                               │
│  ┌─────────┐           ┌─────────────┐                        │
│  │ 스크린   │ ←──────── │ 키입력/마우스│                        │
│  │ 세이버   │           │  출력대기    │                        │
│  └─────────┘           └─────────────┘                        │
│                                                               │
└───────────────────────────────────────────────────────────────┘
```

#### 다. State Diagram 구성요소 `시종상전`

| 구성요소 | 설명 | 표기법 |
|:---------|:-----|:------|
| **시**작점 (Initial State) | 객체의 시작 상태 | ● |
| **종**료점 (Final State) | 객체의 종료 상태 | ◎ |
| **상**태 (State) | 객체의 상태, 내부 전이 포함 | ▭ (둥근 모서리) |
| **전**이 (Transition) | Event[condition]/Action - 객체 상태가 다른 상태로 변경 | → |

#### 라. State Diagram vs Activity Diagram 비교

| 구분 | State Diagram | Activity Diagram |
|:-----|:--------------|:-----------------|
| **관점** | 객체의 상태 변화 | 업무/활동의 흐름 |
| **초점** | 이벤트에 의한 상태 전이 | 제어 흐름과 병렬 처리 |
| **표현** | 하나의 객체 수명주기 | 여러 객체의 상호작용 |
| **용도** | 임베디드, 상태머신 설계 | 비즈니스 프로세스 모델링 |

### 4. 유스케이스 다이어그램 (Use-Case Diagram) 설명

{: .highlight }
> **Use-Case Diagram**: 사용자 관점, 기능 요구사항, 외부요소 표현
> - (정의) `기외사동`
> - (기본구성) `액유시`
> - (관계표현) `연확포일그`

#### 가. Use-Case Diagram의 정의

> **Use-Case Diagram**: 시스템이 제공하고 있는 **기능 및 그와 관련된 외부요소**를 사용자 관점에서 **요구사항을 표현**하는 동적 다이어그램

| 역할 | 설명 |
|:-----|:-----|
| ① 이해관계자 의사소통 | 개발자-기획자-고객 간 공통 언어 |
| ② 요구사항 정의 | 시스템 기능 범위 정의 |
| ③ 기능 범위 식별 | 시스템 경계와 Actor 관계 명확화 |

#### 나. Use-Case Diagram 개념도

```
┌──────────────────────────────────────────────────────────────┐
│                      << System >>                            │
│  ┌────────────────────────────────────────────────────────┐  │
│  │                                                        │  │
│  │       ┌──────────┐          ┌──────────┐              │  │
│  │       │ 상품주문  │◄─────────│ 결제처리  │              │  │
│  │       └──────────┘ include  └──────────┘              │  │
│  │            ▲                      ▲                    │  │
│  │            │                      │                    │  │
│  │  ┌─────────┴──────────┐          │                    │  │
│  │  │                     │          │                    │  │
│  │  │          ┌──────────┴────┐     │                    │  │
│  │  │          │ 배송정보입력   │◄────┘                    │  │
👤───┼──────────►└───────────────┘                          │  │
고객 │  │                                                    │  │
│  │  │          ┌──────────────┐                          │  │
│  │  └──────────│  회원가입     │                          │  │
│  │             └──────────────┘                          │  │
│  │                    ▲                                   │  │
│  │                    │ extend                           │  │
│  │             ┌──────────────┐                          │  │
│  │             │  쿠폰등록     │                          │  │
│  │             └──────────────┘                          │  │
│  └────────────────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────────────────┘
```

#### 다. Use-Case Diagram의 특징

| 특징 | 설명 |
|:-----|:-----|
| **직관적** | 사용자의 기능적 요구사항을 정의하는 직관적인 표현 |
| **사용자와 사용자** | Use Case와 Actor간의 관계를 표현 (Actor는 시스템을 제외한 모든 외부요소로서 시스템을 사용하는 사람 또는 사물, Actor는 Use Case를 수행함) |
| **분석단계** | 주로 분석단계에서 수행하여 시스템 개발 전 단계에 영향 |
| **기능설명** | 시스템이 제공하는 기본적인 기능을 설명 |

#### 라. Use-Case Diagram 기본 구성요소 `액유시`

| 구분 | 구성요소 | 설명 | 표기법 |
|:-----|:---------|:-----|:------|
| **기본구성** | **액**터 (Actor) | 시스템 외부에 존재, 시스템과의 상호작용 주체 | 👤 |
| | **유**스케이스 (Use Case) | 시스템이 제공해야 하는 서비스, Actor가 시스템을 통한 일련의 행위 | (타원) |
| | **시**스템 (System) | 전체시스템의 영역을 표현, 특별한 의미를 가지지 않음 | 사각형 |

#### 마. Use-Case 관계 표현 `연확포일그`

| 관계 | 설명 | 표기법 |
|:-----|:-----|:------|
| **연**관 (Association) | Use Case와 Actor의 관계를 표현 (실선) | ─── |
| **확**장 (Extend) | Optional behavior. 선택적인 행동. extension method의 stereotype으로 표시 (기본 Usecase 수행 시 특별한 조건을 만족할 때 수행하는 Usecase) | - - -▷ `<<extend>>` |
| **포**함 (Include) | Mandatory behavior. 필수적인 행동. extension method의 stereotype으로 표시 (시스템의 기능이 별도의 기능을 포함, Usecase를 수행할 때 다른 Usecase가 반드시 수행되는 것) | - - -▷ `<<include>>` |
| **일**반화 (Generalization) | 하위 Usecase/Action이 상위 Usecase/Actor에게 기능/역할을 상속 받음 | ───▷ |
| **그**룹화 (Grouping) | 여러 개의 Usecase를 단순화 하는 방법 | Package |

### 5. 시스템의 적정 구조 표현, UML 구조 다이어그램

{: .highlight }
> **UML 구조 다이어그램**: 정적 구조, 클래스/객체, 상호보완
> - (종류) `클객패배`
> - (특징) `정속의`

#### 가. 구조 다이어그램 종류

| 구분 | 다이어그램 | 설명 |
|:-----|:----------|:-----|
| **구조** | **클**래스 다이어그램 (Class Diagram) | 클래스의 속성·메서드, 관계 표현 |
| | **객**체 다이어그램 (Object Diagram) | 특정 시점의 객체들과 관계 표현 |
| | **패**키지 다이어그램 (Package Diagram) | 소프트웨어의 구성 요소와 의존성 표현 |
| | **배**치 다이어그램 (Deployment Diagram) | 물리적 하드웨어 구조와 소프트웨어 배치 |

#### 나. 행위 다이어그램과 구조 다이어그램의 상호보완

```
┌──────────────────┐          ┌──────────────────┐
│   구조 다이어그램   │◄────────►│   행위 다이어그램   │
│ (Structure)      │  상호     │  (Behavior)      │
│                  │  보완적   │                  │
│ • 클래스          │  결합     │ • 활동 다이어그램   │
│ • 객체            │          │ • 상태 다이어그램   │
│ • 패키지          │          │ • 유스케이스       │
│ • 배치            │          │ • 시퀀스          │
└──────────────────┘          └──────────────────┘
                    ↓
        ┌─────────────────────────┐
        │ 시스템의 전반적 이해와    │
        │ 설계를 완성              │
        └─────────────────────────┘
```

---

### ⭐ 차별점 키워드

| 다이어그램 | 핵심 차별점 |
|:----------|:----------|
| **Activity Diagram** | Workflow 기반, 병렬 프로세스 지원, Swim Lane으로 역할 구분 |
| **State Diagram** | 단일 객체의 상태 변화, 이벤트-조건-액션 기반 전이 |
| **Use-Case Diagram** | 사용자 관점, 기능 요구사항 정의, Extend/Include 관계 |
| **구조 vs 행위** | 정적 구조와 동적 행위의 상호보완적 결합으로 설계 완성 |

---

### 📈 상위 토픽 계층도

```
UML (Unified Modeling Language)
├── 구조 다이어그램 (Structure Diagram)
│   ├── 클래스 다이어그램 (Class)
│   ├── 객체 다이어그램 (Object)
│   ├── 컴포넌트 다이어그램 (Component)
│   ├── 배치 다이어그램 (Deployment)
│   └── 패키지 다이어그램 (Package)
└── 행위 다이어그램 (Behavior Diagram) ← 현재 토픽
    ├── 활동 다이어그램 (Activity) ← 현재 토픽
    ├── 상태 다이어그램 (State) ← 현재 토픽
    ├── 유스케이스 다이어그램 (Use-Case) ← 현재 토픽
    ├── 시퀀스 다이어그램 (Sequence)
    └── 인터랙션 다이어그램 (Interaction)
```
