# 135회 1교시 5번 모놀리식 아키텍처 vs 마이크로서비스 아키텍처

<details markdown="block">
<summary><strong>🎓 고딩도 이해하는 개념</strong></summary>

**모놀리식 아키텍처**는 피자 한 판처럼 모든 재료가 하나로 합쳐진 구조예요. 만들기는 쉽지만, 페퍼로니만 바꾸고 싶어도 피자 전체를 다시 만들어야 해요.

**마이크로서비스 아키텍처**는 뷔페처럼 각 음식이 따로따로 있는 구조예요. 샐러드가 떨어지면 샐러드만 보충하면 되고, 스테이크 요리사가 휴가 가도 다른 음식은 계속 제공할 수 있어요.

| 비유 | 모놀리식 | 마이크로서비스 |
|---|---|---|
| 음식 | 피자 한 판 | 뷔페 |
| 특징 | 한 번에 다 만듦 | 각각 따로 준비 |
| 수정 | 전체 다시 만듦 | 해당 부분만 수정 |

</details>

---

## 1. 모놀리식 아키텍처와 마이크로서비스 아키텍처의 개념

| 구분 | 개념 |
|---|---|
| **모놀리식 아키텍처** | 하나의 애플리케이션으로 모든 기능/서비스가 단일 코드 베이스로 실행되는 아키텍처 |
| **마이크로서비스 아키텍처** | 개별적인 작은 서비스들이 독립적으로 실행되며, API를 통해 서로 통신하는 아키텍처 |

---

## 2. 모놀리식 vs 마이크로서비스 상세 비교

### 1) 개발/배포 관점 비교

| 비교 항목 | 모놀리식 아키텍처 | 마이크로서비스 아키텍처 |
|---|---|---|
| **개발 복잡도** | 상대적으로 낮음 (초기 개발이 쉬움) | 상대적으로 높음 (분산 시스템 관리 필요) |
| **배포/유지보수** | 작은 변경도 전체 재배포 필요 | 개별 마이크로서비스 단위로 배포 가능 |
| **기술 스택** | 하나의 기술 스택을 사용해야 함 | 서비스별 다른 기술 스택 선택 가능 (폴리글랏) |
| **테스트** | 통합 테스트가 쉬움 | 개별/통합 테스트가 필요해 복잡함 |

### 2) 운영/확장 관점 비교

| 비교 항목 | 모놀리식 아키텍처 | 마이크로서비스 아키텍처 |
|---|---|---|
| **확장성** | 전체 시스템 한 번에 확장으로 비효율적 | 서비스별로 개별 확장 가능 |
| **성능** | 단일 프로세스로 내부 호출 속도가 빠름 | 서비스 간 통신이 필요해 성능 저하 가능 |
| **장애 격리** | 하나의 서비스가 전체에 영향을 미침 | 개별 장애가 미치는 영향을 최소화 |
| **데이터 관리** | 단일 데이터베이스 사용 | 각 서비스별 독립적인 DB 사용 가능 |

### 3) 보안/비용 관점 비교

| 비교 항목 | 모놀리식 아키텍처 | 마이크로서비스 아키텍처 |
|---|---|---|
| **보안** | 내부에서 처리되어 보안 관리가 상대적으로 간단 | 서비스 간 API 통신이 필요하므로 보안 관리 복잡 |
| **운영 비용** | 작은 프로젝트에서는 유지보수 비용이 낮음 | 클라우드 네이티브 환경에서 최적화 시 비용 절감 가능 |

---

## 3. 아키텍처 선택 기준

| 선택 기준 | 모놀리식 | 마이크로서비스 |
|---|---|---|
| **프로젝트 규모** | 소규모/단순 프로젝트 | 대규모/복잡한 프로젝트 |
| **팀 구성** | 소규모 팀 | 다수의 독립적 팀 |
| **변경 빈도** | 변경이 적은 시스템 | 빈번한 배포 필요 시 |
| **확장 요구** | 균일한 확장 | 서비스별 다른 확장 요구 |

---

**암기법:**
- (모놀리식 특징) `단통간장` (단일코드, 통합배포, 간단보안, 장애전파)
- (마이크로서비스 특징) `분독폴격` (분산시스템, 독립배포, 폴리글랏, 격리된장애)
- (비교항목) `개배확성장데보운` (개발복잡도, 배포, 확장성, 성능, 장애격리, 데이터, 보안, 운영비용)
