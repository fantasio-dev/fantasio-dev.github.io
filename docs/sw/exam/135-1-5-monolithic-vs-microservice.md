### 🔑 핵심 키워드 3개

| 키워드 | 설명 |
|:------|:-----|
| **모놀리식** | 모든 기능이 하나로 합쳐진 단일 애플리케이션 구조 |
| **마이크로서비스** | 작은 서비스들이 독립적으로 실행되며 API로 통신하는 구조 |
| **폴리글랏** | 서비스별로 다른 기술 스택을 선택할 수 있는 특성 |

---

### 📖 등장배경

```
🌐 과거: 모든 기능을 하나의 애플리케이션으로 개발 (모놀리식)

💡 문제: "서비스가 커지니까 작은 수정에도 전체를 재배포해야 하고, 
         한 곳의 장애가 전체 시스템을 멈추게 하네?"

✅ 해결: 마이크로서비스 아키텍처로 해결!
   → 서비스별 독립 배포 가능
   → 장애 격리로 시스템 안정성 향상
   → 각 서비스에 최적화된 기술 스택 선택 가능
```

---

### 📝 개념 정의

> **모놀리식**: 단일 코드 베이스로 모든 기능이 실행되는 아키텍처
> **마이크로서비스**: 독립 서비스들이 API로 통신하는 분산 아키텍처

### 쉬운 비유

| 모놀리식 | 마이크로서비스 |
|:------|:------|
| 🍕 피자 한 판 (토핑 하나만 바꿔도 전체 다시 만들어야 함) | 🍽️ 뷔페 (샐러드만 보충하면 됨) |
| 📦 하나의 큰 상자 | 📦📦📦 여러 개의 작은 상자 |

---

### 🏗️ 기술요소

#### 그룹 1: 모놀리식 특징 `단통간장`

| 항목 | 설명 | 예시 |
|:-----|:-----|:-----|
| **단**일 코드 | 하나의 코드 베이스 | 단일 WAR/JAR 배포 |
| **통**합 배포 | 작은 변경도 전체 재배포 | 전체 빌드 후 배포 |
| **간**단 보안 | 내부 호출로 보안 관리 간단 | 단일 인증 체계 |
| **장**애 전파 | 한 곳 장애가 전체에 영향 | 메모리 누수 시 전체 다운 |

#### 그룹 2: 마이크로서비스 특징 `분독폴격`

| 항목 | 설명 | 예시 |
|:-----|:-----|:-----|
| **분**산 시스템 | 여러 서비스로 분산 | 주문/결제/배송 서비스 분리 |
| **독**립 배포 | 서비스 단위 개별 배포 | 결제 서비스만 업데이트 |
| **폴**리글랏 | 서비스별 다른 기술 스택 | Java, Node.js, Python 혼용 |
| **격**리된 장애 | 장애가 해당 서비스에만 영향 | Circuit Breaker 패턴 |

---

### ⭐ 차별점 키워드

**독립 배포 vs 전체 배포**: MSA는 서비스 단위 독립 배포, 모놀리식은 전체 재배포 필요

## 🎯 기술사 수준 설명

{: .highlight }
> **모놀리식 vs MSA**: 단일 vs 분산, 전체배포 vs 독립배포, 장애전파 vs 장애격리
> - (모놀리식) `단통간장` (단일코드, 통합배포, 간단보안, 장애전파)
> - (마이크로서비스) `분독폴격` (분산시스템, 독립배포, 폴리글랏, 격리된장애)
> - (비교항목) `개배기테 확성장데 보운규팀` (12개)
> - ⭐ **차별점**: MSA는 독립 배포 + 폴리글랏 + 장애 격리, 모놀리식은 단순성 + 내부 호출 속도

### 🔑 핵심 키워드 3개

| 키워드 | 설명 |
|:------|:-----|
| **모놀리식 아키텍처** | 모든 기능이 단일 코드 베이스로 실행되는 일체형 아키텍처 |
| **마이크로서비스 아키텍처** | 독립적인 작은 서비스들이 API로 통신하는 분산 아키텍처 |
| **폴리글랏 아키텍처** | 서비스별로 최적의 기술 스택을 선택할 수 있는 구조 |

### 📖 등장배경

```
🌐 과거: 대부분의 시스템이 모놀리식으로 개발됨 (단순, 관리 용이)

💡 문제: 서비스 규모 확대, 빠른 배포 주기, 클라우드 환경 필요

✅ 해결: 마이크로서비스 아키텍처 도입
   → 서비스별 독립 배포로 민첩성 향상
   → 장애 격리로 시스템 안정성 확보
   → 클라우드 네이티브 환경에 최적화
```

| **마이크로서비스 아키텍처** | 독립적 서비스들이 API 통신으로 협력하는 분산 아키텍처 |

### 🏗️ 구성요소

#### 그룹 1: 모놀리식 구성요소 `단데공통`

| 항목 | 설명 | 예시 |
|:-----|:-----|:-----|
| **단**일 코드 베이스 | 모든 기능이 하나의 코드로 통합 | 단일 Spring Boot 애플리케이션 |
| **데**이터베이스 공유 | 하나의 DB를 전체가 공유 | 단일 Oracle/MySQL |
| **공**통 기술 스택 | 동일한 언어/프레임워크 사용 | Java + Spring 전체 적용 |
| **통**합 배포 단위 | 전체가 하나의 배포 단위 | 단일 WAR/JAR 파일 |

#### 그룹 2: 마이크로서비스 구성요소 `서게분독`

| 항목 | 설명 | 예시 |
|:-----|:-----|:-----|
| **서**비스 분리 | 비즈니스 도메인별 서비스 분리 | 주문/결제/배송 서비스 |
| **게**이트웨이 | API Gateway를 통한 진입점 통합 | Kong, AWS API Gateway |
| **분**산 데이터 | 서비스별 독립 데이터베이스 | 각 서비스별 DB |
| **독**립 배포 | 서비스 단위 개별 배포 | Docker + Kubernetes |

### 📊 모놀리식 vs 마이크로서비스 상세 비교

#### 비교 구분자 `개배기테 확성장데 보운규팀`

| 비교 항목 | 모놀리식 아키텍처 | 마이크로서비스 아키텍처 |
|:---------|:-----------------|:----------------------|
| **개**발 복잡도 | 상대적으로 낮음 (초기 개발 용이) | 상대적으로 높음 (분산 시스템 관리) |
| **배**포 방식 | 작은 변경도 전체 재배포 필요 | 서비스 단위 독립 배포 가능 |
| **기**술 스택 | 단일 기술 스택 사용 | 서비스별 다른 기술 스택 (폴리글랏) |
| **테**스트 | 통합 테스트 용이 | 개별/통합 테스트 복잡 |
| **확**장성 | 전체 시스템 스케일링 (비효율) | 서비스별 개별 스케일링 |
| **성**능 | 단일 프로세스 내부 호출 (빠름) | 서비스 간 네트워크 통신 (오버헤드) |
| **장**애 격리 | 한 곳 장애가 전체에 영향 | 개별 장애 영향 최소화 |
| **데**이터 관리 | 단일 데이터베이스 공유 | 서비스별 독립 DB (Polyglot Persistence) |
| **보**안 | 내부 처리로 간단 | API 통신 보안 관리 복잡 |
| **운**영 비용 | 소규모에서 유지보수 비용 낮음 | 클라우드 최적화 시 비용 절감 |
| **규**모 적합성 | 소규모/단순 프로젝트 | 대규모/복잡한 프로젝트 |
| **팀** 구성 | 소규모 단일 팀 | 다수의 독립적 팀 (Conway's Law) |

### 📝 MSA 답안용 키워드

| 키워드 | 설명 |
|:------|:-----|
| **API Gateway** | 단일 진입점, 라우팅, 인증 |
| **Service Mesh** | 서비스 간 통신 관리 (Istio, Linkerd) |
| **Circuit Breaker** | 장애 전파 방지 패턴 |
| **Event-Driven** | 비동기 메시지 기반 통신 |
| **Container Orchestration** | Kubernetes 기반 배포 관리 |

---

### ✨ 기술사 답안 차별화 키워드

> **Strangler Fig Pattern**
> 
> 모놀리식에서 MSA로 점진적 전환을 위한 패턴
> 
> * 기존 모놀리식을 감싸는 형태로 새 서비스 구축
> * 점진적으로 기능을 MSA로 이전
> * 레거시 시스템의 안전한 현대화
