### 🔑 핵심 키워드 3개

| 키워드 | 설명 |
|:------|:-----|
| **점진적 개발** | 한 번에 다 만들지 않고 조금씩 완성해나가는 방식 |
| **빠른 피드백** | 고객에게 자주 보여주고 의견을 바로 반영 |
| **유연한 대응** | 요구사항이 바뀌어도 유연하게 대처 가능 |

---

### 📖 등장배경

🌐 과거: 폭포수 모델처럼 계획을 세우고 순서대로 개발했는데...

💡 문제: "고객이 원하는 게 자꾸 바뀌는데 어떻게 해요?"

✅ 해결: 작은 단위로 빠르게 개발하자!
   → 짧은 주기(스프린트)로 개발
   → 고객 피드백을 자주 받음
   → 변화에 유연하게 대응!

---

### 📝 개념 정의

> **애자일(Agile)**: 절차보다 사람과 제품에 집중하여 고객 요구에 유연하게 대응하는 개발 방법론

### 쉬운 비유

| 폭포수(Waterfall) | 애자일(Agile) |
|:-----------------|:--------------|
| 요리 레시피대로 정확히 만들기 | 맛보면서 간 조절하며 만들기 |
| 계획대로만! 변경 어려움 | 상황 봐가며 유연하게! |

### 🏗️ 기술요소

#### 그룹 1: 장점 `고창생품`

| 장점 | 설명 |
|:-----|:-----|
| **고**객 가치 | 빠르게 가치있는 기능 전달 |
| **창**의성 | 팀 자율성으로 업무 만족도 UP |
| **생**산성 | 불필요한 문서 줄이고 협업 강화 |
| **품**질 향상 | 자주 테스트하고 피드백 반영 |

#### 그룹 2: 단점 `문요사감`

| 단점 | 설명 |
|:-----|:-----|
| **문**서화 부족 | 체계적인 지침이 없어서 혼란 |
| **요**구사항 오버헤드 | 자주 바뀌면 테스트도 다시 |
| **사**업 관리 미흡 | 상위 관리자 보고용 지표 부족 |
| **감**리 대응 어려움 | 문서가 적어서 감리 기준 통과 힘듦 |

**"변화 대응력"** - 요구사항 변경을 비용으로 보지 않고 기회로 봄

## 🎯 기술사 수준 설명

{: .highlight }
> **애자일(Agile)**: 절차보다 사람 중심 → 점진적 상세화, 고객 피드백, 유연한 대응 → 낭비 제거하고 가치 전달하는 개발 방법론
> - (장점) `고창생품` (고객가치, 창의성, 생산성, 품질)
> - (단점) `문요사감` (문서화, 요구사항, 사업관리, 감리)
> - (유형) `스칸XP` (스크럼, 칸반, XP)
> - ⭐ **차별점**: 요구사항 변경을 비용이 아닌 기회로 수용

---

### 🔑 핵심 키워드 3개

| 키워드 | 설명 |
|:------|:-----|
| **점진적 상세화** | 반복(Iteration)을 통해 점진적으로 기능 완성 |
| **고객 협력** | 계약 협상보다 고객과의 협력 중시 |
| **변화 대응** | 계획 따르기보다 변화에 대응 중시 |

---

### 📖 등장배경

🌐 과거: 폭포수(Waterfall) 모델 중심의 계획 기반 개발

💡 문제: "요구사항 변경 시 비용 증가, 늦은 고객 피드백으로 실패 프로젝트 증가"

✅ 해결: 2001년 애자일 선언(Agile Manifesto) 발표
   → 개인과 상호작용 > 프로세스와 도구
   → 동작하는 소프트웨어 > 포괄적인 문서
   → 고객과 협력 > 계약 협상
   → 변화에 대응 > 계획 따르기

---

### 📝 개념 정의

> **애자일(Agile)**: 효율적인 제품, SW 개발을 위해 절차보다는 사람과 제품에 집중하여 낭비를 제거하고 고객의 요구사항을 보다 유연하고 신속하게 반영하기 위한 방법론

---

### 🏗️ 구성요소

#### 그룹 1: 장점 `고창생품`

| 항목 | 설명 | 예시 |
|:-----|:-----|:-----|
| **고**객 가치(ROI 증대) | 가치 있는 기능을 빠르고 안정적으로 전달 | Time to Market 단축 |
| **창**의성 향상 | 팀 자율성 강화를 통한 업무만족도 향상 | 자기조직화 팀 운영 |
| **생**산성 향상 | 불필요한 산출물 제거 및 팀 협업 강화 | 최소 문서화, 페어 프로그래밍 |
| **품**질 향상 | 고객의 주기적 피드백 및 빈번한 테스트 | TDD, CI/CD |

#### 그룹 2: 단점 `문요사감`

| 항목 | 설명 | 예시 |
|:-----|:-----|:-----|
| **문**서화 부족 | Agile 프로세스 적용 지원 문서/지침 부족 | 통합/테스트 가이드 미흡 |
| **요**구사항 오버헤드 | 비기능 요구사항 고려 부족, 잦은 변경에 따른 테스트 증가 | 리그레션 테스트 비용 |
| **사**업관리 미흡 | 측정지표가 개발자 위주, 리스크 관리 고려 부족 | 상위 보고 지표 부재 |
| **감**리 대응 문제 | 적은 문서량, 감리 기준 부합 어려움 | 산출물 이슈 발생 |

---

### 🔄 주요 애자일 유형 비교

| 유형 | 장점 | 단점 |
|:-----|:-----|:-----|
| **스크럼(Scrum)** | 짧은 스프린트로 빠른 피드백 | 고정된 스프린트로 유연성 부족 |
| **칸반(Kanban)** | 작업 흐름 시각적 관리 | 장기 계획 수립 어려움 |
| **XP** | 유연한 계획 수립, 기술 프랙티스 | 높은 고객 참여 요구 |

---

### ⭐ 차별점 키워드

| 차별점 | 설명 |
|:------|:-----|
| **변화 수용** | 요구사항 변경을 비용이 아닌 **기회**로 수용 |
| **점진적 상세화** | 초기에 모든 것을 정하지 않고 **반복적으로 상세화** |
| **고객 중심** | 개발팀과 고객이 **지속적으로 협업** |
