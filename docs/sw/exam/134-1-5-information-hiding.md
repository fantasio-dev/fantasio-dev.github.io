### 📖 등장배경

🌐 과거: 객체지향 프로그래밍이 등장하면서 여러 객체가 서로 데이터를 주고받기 시작했어요.

💡 문제: "다른 객체가 내 데이터를 마음대로 바꾸면 어떡하지? 버그가 생기면?"

✅ 해결: 정보은닉으로 해결!
   → 내부 데이터는 숨기고
   → 공개된 인터페이스로만 소통하게 하고
   → 데이터 무결성과 보안을 지켜요!

---

### 📝 개념 정의

> **정보은닉**: 내부 데이터를 숨기고 공개 인터페이스로만 접근하게 하는 기법

### 쉬운 비유 🏠

| 개념 | 비유 |
|:-----|:-----|
| 정보은닉 | 집의 **대문**! 집 안의 물건(데이터)은 숨기고, 방문자는 대문(인터페이스)을 통해서만 들어와요 |

```
👤 외부 객체      🚪 공개 인터페이스      📦 내부 데이터
   (손님)    →     (대문/현관)     →    (집안 물건)
                    ↑
            여기서 접근 제어!
```

### 🏗️ 기술요소 (2그룹)

#### 그룹 1: 접근제어자 `퍼프프패`

| 구분 | 접근 범위 |
|:-----|:----------|
| **퍼**blic | 모든 클래스, 패키지에서 접근 가능 |
| **프**rivate | 해당 클래스 내에서만 접근 가능 |
| **프**rotected | 동일 패키지 내 + 상속받은 클래스 |
| **패**키지(default) | 동일 패키지 내에서만 접근 가능 |

#### 그룹 2: 정보은닉 구현 기법 `상캡인추`

| 기법 | 역할 |
|:-----|:-----|
| **상**위 타입 캐스팅 | 구체적인 타입을 감추고 상위 타입으로 은닉 |
| **캡**슐화 | 데이터와 메서드를 하나로 묶어 보호 |
| **인**터페이스 | 내부 구현을 숨기고 공개 API만 노출 |
| **추**상화 | 세부 구현을 추상 클래스로 분리 |

---

### ⭐ 차별점 키워드

**캡슐화와 차이점**: 캡슐화는 "묶는 것", 정보은닉은 "숨기는 것"!
- 캡슐화: 데이터와 메서드를 클래스로 묶음
- 정보은닉: 그 중 일부를 외부에서 못 보게 숨김

## 🎯 기술사 수준 설명

{: .highlight }
> **정보은닉**: 복잡성 제거 + 데이터 보호 → 내부 구현을 숨기고 공개 인터페이스로만 접근하게 하는 코드 보안 기술
> - (접근제어자) `퍼프프패`
> - (구현기법) `상캡인추`
> - ⭐ **차별점**: 캡슐화가 "묶는 것"이라면, 정보은닉은 "숨기는 것"

### 🏗️ 구성요소

#### 그룹 1: 접근제어자 `퍼프프패`

| 항목 | 설명 | 예시 |
|:-----|:-----|:-----|
| **퍼**blic | 모든 클래스, 패키지에서 접근 가능 | `public void display()` |
| **프**rivate | 해당 클래스 내에서만 접근 가능, 최소 권한 | `private int balance;` |
| **프**rotected | 동일 패키지 + 상속 시 타 패키지에서 접근 | `protected void init()` |
| **패**키지(default) | 동일 패키지 내에서만 접근 가능 | `void process()` |

#### 그룹 2: 정보은닉 구현 기법 `상캡인추`

| 항목 | 설명 | 예시 |
|:-----|:-----|:-----|
| **상**위 타입 캐스팅 | 객체의 구체적인 타입을 상위 타입으로 은닉 | `Animal a = new Dog();` |
| **캡**슐화 | 객체의 필드 및 메서드를 하나로 묶어 보호 | getter/setter 패턴 |
| **인**터페이스 | 객체 내부 구현을 숨기고 공개 API만 노출 | `interface Payment { void pay(); }` |
| **추**상화 | 세부 구현을 추상 클래스/메서드로 분리 | `abstract class Shape` |

### 💻 샘플코드

```java
public class BankAccount {
    // private: 외부에서 직접 접근 불가 (정보은닉)
    private String accountNumber;
    private double balance;

    // public: 외부에서 접근 가능한 인터페이스
    public double getBalance() {
        return this.balance;
    }

    public void deposit(double amount) {
        if (validateAmount(amount)) {
            this.balance += amount;
        }
    }

    public boolean withdraw(double amount) {
        if (validateAmount(amount) && this.balance >= amount) {
            this.balance -= amount;
            return true;
        }
        return false;
    }

    // private: 내부에서만 사용하는 유효성 검증
    private boolean validateAmount(double amount) {
        return amount > 0;
    }
}
```

### ⭐ 차별점 키워드 (상세)

| 구분 | 캡슐화(Encapsulation) | 정보은닉(Information Hiding) |
|:-----|:-----|:-----|
| **정의** | 데이터와 메서드를 하나의 단위로 묶음 | 내부 구현을 외부로부터 숨김 |
| **목적** | 관련 기능의 응집도 향상 | 데이터 보호 및 복잡성 제거 |
| **수단** | 클래스 정의 | 접근제어자, 인터페이스 |
| **비유** | 약을 캡슐에 담는 것 | 캡슐 안의 성분을 숨기는 것 |

> **핵심 포인트**: 캡슐화는 정보은닉의 전제조건! 먼저 묶어야(캡슐화) 숨길 수(정보은닉) 있다.
