### 🔑 핵심 키워드 3개

| 키워드 | 설명 |
|:------|:-----|
| **임시 방편** | 빠르게 끝내려고 대충 만든 코드 |
| **누적 비용** | 나중에 갚아야 할 기술적 빚 |
| **품질 저하** | 시간이 지날수록 코드가 엉망이 됨 |

---

### 📖 등장배경

```
🌐 과거: 빠른 출시를 위해 대충 코딩하는 일이 많았음

💡 문제: "임시로 만든 코드가 쌓이면 나중에 어떻게 될까?"

✅ 해결: 기술 부채라는 개념으로 인식하고 관리!
   → 부채처럼 이자(추가 비용)가 붙는다는 것을 인지
   → 정기적으로 코드 정리(리팩토링) 필요
   → 품질과 일정의 균형 유지
```

---

### 📝 개념 정의

> **기술 부채**: 빠른 개발을 위해 임시 방편을 선택해 발생하는 추가 비용

| 구분 | 정의 (22자 이내) |
|:-----|:----------------|
| **기술 부채** | 임시 해법으로 인한 미래 비용 |
| **부채 이자** | 시간이 지날수록 증가하는 수정 비용 |

### 쉬운 비유

| 신용카드 빚 | 기술 부채 |
|:-----|:-----|
| 돈 빌려 지금 당장 물건 구매 | 대충 코딩해서 빨리 출시 |
| 이자 붙어서 더 많이 갚아야 함 | 나중에 고치려면 더 큰 비용 발생 |

---

### 🏗️ 기술요소

#### 그룹 1: 부채 유형 `설코테문`

| 유형 | 설명 | 예시 |
|:-----|:-----|:-----|
| **설**계 부채 | 모듈성 부족, 규칙 위반 | 결합도 높은 코드 |
| **코**드 부채 | 정적 분석 위반, 컨벤션 무시 | 리뷰 생략된 코드 |
| **테**스트 부채 | 방법론 부재, 커버리지 부족 | QA 계획서 없음 |
| **문**서 부채 | 산출물 누락, 최신화 안됨 | 화면설계서 없음 |

#### 그룹 2: 관리 방법 `추우템기리`

| 방법 | 설명 | 예시 |
|:-----|:-----|:-----|
| **추**정 | 부채 식별, 정적 분석 | SonarQube |
| **우**선순위화 | 리스크 기반 정렬 | 정량/정성 평가 |
| **템**플릿 통합 | 코드 표준화 도구 적용 | Prettier, Lint |
| **기**술 업데이트 | 오래된 기술 최신화 | Docker, K8s |

---

### ⭐ 차별점 키워드

> **"빚처럼 이자가 붙는다"** - 미루면 미룰수록 수정 비용이 기하급수적으로 증가

## 🎯 기술사 수준 설명

{: .highlight }
> **기술 부채**: 임시 해법 선택 → 설계/코드/테스트/문서 부채 → 차후 추가 비용 위험
> - (유형) `설코테문`
> - (관리) `추우템기리`
> - ⭐ **차별점**: 품질 투자비용과 기술 부채 간 Balance 유지가 핵심

---

### 📝 개념 정의

#### 🔑 핵심 키워드 3개

| 키워드 | 설명 |
|:------|:-----|
| **임시 방편** | 장기 솔루션 대신 단기 해법 선택 |
| **누적 비용** | 시간 경과에 따라 증가하는 수정 비용 |
| **품질 저하** | SW 안전성 및 유지보수성 하락 |

---

#### 📖 등장배경

```
🌐 과거: Time-To-Market 압박으로 무리한 일정 진행

💡 문제: "임시 코드가 누적되면 기술 파산 위험이 발생하지 않을까?"

✅ 해결: 기술 부채 개념 도입으로 체계적 관리!
   → 부채 유형별 분류 및 식별
   → 우선순위 기반 해결
   → 품질과 일정의 균형 유지
```

---

#### 📝 기술 부채의 정의

> **소프트웨어 기술 부채**: 설계·개발·테스트·배포 전 과정에서 장기적 솔루션 대신 임시 방편을 선택해 발생하는 추가 위험 비용

| 구분 | 정의 |
|:-----|:-----|
| **기술 부채** | 차후 발생 가능한 추가적 위험 비용 |
| **기술 파산** | 기술 부채 누적으로 시스템 유지 불가 상태 |

---

#### 📊 기술 부채의 원인

| 원인 | 설명 |
|:-----|:-----|
| **Business Pressure** | Time-To-Market 무리한 일정, 적은 예산 |
| **Low Technology Maturity** | 잘못된 아키텍처, 개발자 경험·협업 부족 |
| **Frequent Requirement Changes** | 코드 복잡성 상승, 낮은 테스트 커버리지 |
| **Lack of Experts** | 부족한 테스팅·문서화, 스파게티 코드 |

---

### 🏗️ 구성요소

#### 그룹 1: 기술 부채 유형 `설코테문`

| 유형 | 핵심 기술 | 설명 |
|:-----|:-----|:-----|
| **설**계 부채 | 설계 악취, 규칙 위반 | 모듈성 부족(결합도↑, 응집도↓), 잦은 규칙 위반 |
| **코**드 부채 | 정적 분석 위반, 컨벤션 위반 | Code Review 생략, Lint/Formatter 미활용 |
| **테**스트 부채 | 방법론 부재, 커버리지 부족 | DevOps/Agile 테스트 방법론 부재, QA 계획서 부재 |
| **문**서 부채 | 산출물 누락, 신기술 문서 부재 | 화면설계서/DB설계서 누락, 최신 문서화 누락 |

#### 그룹 2: 관리 방법 `추우템기리`

| 관리 방법 | 설명 | 기법·도구 |
|:-----|:-----|:-----|
| **추**정 (부채 추정) | 부채 식별, 기술적 부채 유형 검토, 코드 정적 분석 | Coverity, SonarQube, CheckStyle |
| **우**선순위화 | 이슈/리스크 측정 기반 우선순위화, 높은 Task부터 해결 | 정량적 평가, 정성적 평가 |
| **템**플릿 통합 | 모든 코드에 템플릿 도구 적용, 표준화 | Prettier, Lint |
| **기**술 업데이트 | 오래된 기술 최신화, 오픈소스 활용 | 컨테이너, Docker, K8s |

---

### 🔄 리팩토링과 테스팅

| 활동 | 설명 | 기법 |
|:-----|:-----|:-----|
| **리팩토링** | 모듈성 결여 코드 개선 | 응집도 높임, 결합도 낮춤 |
| **코드 리뷰** | 동료 검토 수행 | Walk Through, Peer Review, Inspection |
| **테스팅** | 테스트 방법론 도입 | Black Box Test, White Box Test |

---

### ⭐ 차별점 키워드

| 차별점 | 설명 |
|:------|:-----|
| **품질-부채 Balance** | 품질 투자비용과 기술 부채 간 균형 유지가 핵심 |
| **이자 개념** | 미루면 미룰수록 수정 비용이 기하급수적 증가 |
| **전 과정 적용** | 설계·개발·테스트·배포 전 영역에서 발생 |
