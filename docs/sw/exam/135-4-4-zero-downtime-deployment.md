### 💡 한 줄 정의

> **무중단 배포**: 서비스가 중단되지 않은 상태(Zero-downtime)로 새로운 버전을 사용자에게 배포하는 기술

### 🎭 쉬운 비유

| 배포 방식 | 비유 |
|:---------|:-----|
| **롤링 배포** | 건물 층별 리모델링 - 한 층씩 순차적으로 공사 |
| **블루-그린 배포** | 새 건물 완공 후 이사 - 한 번에 전환 |
| **카나리 배포** | 시험 오픈 후 점진적 확대 - 소수 고객 먼저 테스트 |

### ❓ 왜 필요한가? (3줄 요약)

1. **사용자 경험 향상**: 중단 없는 서비스 사용을 통한 사용자 경험/만족도 향상
2. **비즈니스 연속성 보장**: 중단 없이 배포 가능하므로 서비스 운영 영향 없음
3. **빠른 피드백**: 새로운 버전 배포하면서 실시간 피드백 및 문제 대응 가능

## 🎯 기술사 수준 설명

{: .highlight }
> **무중단 배포**: 롤링, 블루-그린, 카나리 → 서비스 중단 없이 새 버전 배포하는 기술
> - (1) `다유롤` (중단배포 문제점과 무중단 배포 필요성)
> - (2) `롤블카` (무중단 배포 기법의 종류)
> - (3) `리로` (무중단 배포의 기술요소)
> - ⭐ **차별점**: 리버스 프록시와 로드밸런싱 기술을 활용해 무중단 배포 구현

---

### 1. 소프트웨어 중단 배포의 문제점 및 무중단 배포의 필요성

{: .highlight }
> **중단배포 vs 무중단배포**: 다운타임, 유연성, 롤백
> - (문제점) `다유롤`
> - (필요성) `사비빠`

#### 가. 소프트웨어 중단배포의 문제점 `다유롤`

| 구분 | 문제점 | 설명 |
|:-----|:------|:-----|
| **SW 배포의 개요** | 중단배포 vs 무중단배포 | 소프트웨어 개발 이후 결과물을 고객에게 제공하는 방법으로, 시스템을 중단하는 중단배포와 시스템의 중단 없이 배포하는 무중단 배포로 분류 |
| **중단배포 문제점** | **다**운타임 발생 | 24시간 항상 가동되어야 하는 서비스의 경우 문제 발생가능 |
| | **유**연한 개발 불가 | CI/CD등 유연한 개발 및 배포가 불가 |
| | **롤**-Back시 중단 | 배포시 문제 발생할 경우, Rollback시에도 다운타임 발생 |

#### 나. 무중단 배포의 필요성 `사비빠`

| 구분 | 필요성 | 설명 |
|:-----|:------|:-----|
| **무중단 배포** | 정의 | 서비스가 중단되지 않은 상태(Zero-downtime)로 새로운 버전을 사용자에게 배포하는 기술 |
| **무중단 필요성** | **사**용자 경험 향상 | 중단 없는 서비스 사용을 통한 사용자 경험/만족도 향상 |
| | **비**즈니스 연속성 보장 | 중단 없이 배포 가능하므로 서비스 운영 영향 없음 |
| | **빠**른 피드백 | 새로운 버전 배포하면서 실시간 피드백 및 문제 대응 가능 |

---

### 2. 무중단 배포 기법의 종류 설명

{: .highlight }
> **무중단 배포 기법**: 롤링, 블루-그린, 카나리
> - (종류) `롤블카`
> - (핵심기술) `리로` (리버스 프록시, 로드밸런싱)

#### 가. 롤링 배포 (Rolling Deployment)

| 구분 | 설명 |
|:-----|:-----|
| **개념도** | 단계 1(배포 전): 모든 인스턴스 Old Version<br>→ 단계 2: 일부 인스턴스 New Version 교체<br>→ 단계 3: 점진적 교체 진행<br>→ 단계 4(배포 후): 모든 인스턴스 New Version |
| **개념** | 롤링 배포는 **사용 중인 인스턴스 내에서 새 버전을 점진적으로 교체**하는 것으로 무중단 배포의 가장 기본적인 방식 |
| **방식** | 서비스 중인 인스턴스 하나를 로드밸런서에서 라우팅하지 않도록 한 뒤, 새 버전을 적용하여 다시 라우팅하고, 이를 반복하여 모든 인스턴스에 새 버전의 애플리케이션을 배포 |

```
┌─────────────────────────────────────────────────────────────────────┐
│                    Rolling Deployment 개념도                         │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│   단계 1 (배포 전)        단계 2                단계 3               │
│   ┌───┐ ┌───┐ ┌───┐     ┌───┐ ┌───┐ ┌───┐    ┌───┐ ┌───┐ ┌───┐   │
│   │Old│ │Old│ │Old│     │New│ │Old│ │Old│    │New│ │New│ │Old│   │
│   └───┘ └───┘ └───┘     └───┘ └───┘ └───┘    └───┘ └───┘ └───┘   │
│                                                                      │
│   단계 4 (배포 후)                                                   │
│   ┌───┐ ┌───┐ ┌───┐                                                 │
│   │New│ │New│ │New│     ← 모든 인스턴스 새 버전으로 교체 완료        │
│   └───┘ └───┘ └───┘                                                 │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

| 장점 | 단점 |
|:-----|:-----|
| 추가 인프라 비용 없음 | 배포 중 두 버전 혼재 |
| 점진적 배포로 리스크 분산 | 롤백 시간이 오래 걸림 |

#### 나. 블루-그린 배포 (Blue-Green Deployment)

| 구분 | 설명 |
|:-----|:-----|
| **개념도** | 배포 전: Blue(구버전)에 트래픽<br>→ 배포 후: Green(신버전)으로 트래픽 전환 |
| **개념** | 블루를 구버전, 그린을 신버전으로 지칭하여 붙여진 이름으로 **운영 환경에 구버전과 동일하게 신버전의 인스턴스를 구성한 후, 로드밸런서를 통해 신버전으로 모든 트래픽을 전환**하는 배포 방식 |
| **방식** | - 구버전과 동일한 운영 환경으로 신버전의 인스턴스를 구성<br>- 실제 서비스 환경에서 신버전을 미리 테스트 할 수 있는 장점 및 롤링 배포와 마찬가지로 빠른 롤백이 가능<br>- 배포가 완료된 후 남아 있는 기존 버전의 환경을 다음 배포에 재사용 가능 |

```
┌─────────────────────────────────────────────────────────────────────┐
│                   Blue-Green Deployment 개념도                       │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│           배포 전                          배포 후                   │
│                                                                      │
│        ┌───────────┐                    ┌───────────┐               │
│        │  Client   │                    │  Client   │               │
│        └─────┬─────┘                    └─────┬─────┘               │
│              │                                │                      │
│              ▼                                ▼                      │
│     ┌─────────────────┐              ┌─────────────────┐            │
│     │  Load Balancer  │              │  Load Balancer  │            │
│     └────┬───────┬────┘              └────┬───────┬────┘            │
│          │       │                        │       │                  │
│          ▼       ▼                        ▼       ▼                  │
│     ┌────────┐ ┌────────┐           ┌────────┐ ┌────────┐           │
│     │  Blue  │ │ Green  │           │  Blue  │ │ Green  │           │
│     │  (Old) │ │ (New)  │           │  (Old) │ │ (New)  │           │
│     │ ████   │ │        │           │        │ │ ████   │           │
│     │ ████   │ │ ████   │           │ ████   │ │ ████   │           │
│     │ ████   │ │ ████   │           │ ████   │ │ ████   │           │
│     └────────┘ └────────┘           └────────┘ └────────┘           │
│       트래픽 ▲                                   트래픽 ▲            │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

| 장점 | 단점 |
|:-----|:-----|
| 즉시 롤백 가능 | 2배의 인프라 비용 필요 |
| 운영 환경에서 사전 테스트 가능 | 데이터베이스 스키마 변경 시 복잡 |

#### 다. 카나리 배포 (Canary Deployment)

| 구분 | 설명 |
|:-----|:-----|
| **개념도** | 90% Old / 10% New → 10% Old / 90% New → 100% New (단계적 전환) |
| **개념** | **신버전의 제공 범위를 늘려가면서 모니터링 및 피드백 과정을 통해 잠재적 문제 상황을 미리 발견**하기 위한 방식 |
| **방식** | - 신버전의 배포 전에 실제 운영 환경에서 미리 테스트한다는 점이 블루-그린 배포와 비슷한 방식<br>- **단계적인 전환 방식**을 통해 부정적 영향을 최소화하고 상황에 따라 트래픽 양을 늘리거나 롤백 가능<br>- 롤링 배포와 마찬가지로 신·구 두 버전이 운영되기 때문에 버전 관리가 필요 |

```
┌─────────────────────────────────────────────────────────────────────┐
│                    Canary Deployment 개념도                          │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│     단계 1            단계 2            단계 3                       │
│   ┌─────────┐       ┌─────────┐       ┌─────────┐                   │
│   │ Client  │       │ Client  │       │ Client  │                   │
│   └────┬────┘       └────┬────┘       └────┬────┘                   │
│        │                 │                 │                         │
│        ▼                 ▼                 ▼                         │
│  ┌───────────┐     ┌───────────┐     ┌───────────┐                  │
│  │Load Balancer│    │Load Balancer│    │Load Balancer│               │
│  │ 90%   10% │     │ 10%   90% │     │  0%  100% │                  │
│  └──┬─────┬──┘     └──┬─────┬──┘     └──┬─────┬──┘                  │
│     │     │           │     │           │     │                      │
│     ▼     ▼           ▼     ▼           ▼     ▼                      │
│  ┌─────┐ ┌─────┐   ┌─────┐ ┌─────┐   ┌─────┐ ┌─────┐                │
│  │ Old │ │ New │   │ Old │ │ New │   │     │ │ New │                │
│  │████ │ │ █   │   │ █   │ │████ │   │     │ │████ │                │
│  └─────┘ └─────┘   └─────┘ └─────┘   └─────┘ └─────┘                │
│                                                                      │
│     10% 테스트       점진적 확대        100% 전환 완료               │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

| 장점 | 단점 |
|:-----|:-----|
| 점진적 전환으로 리스크 최소화 | 두 버전 병행으로 버전 관리 필요 |
| 실시간 모니터링 및 롤백 가능 | 트래픽 분배 로직 복잡 |

- 리버스 프록시와 로드밸런싱 기술을 활용해 무중단 배포가 가능하도록 구현 가능

### 3. 무중단 배포의 기술요소, 리버스 프록시와 로드밸런싱

{: .highlight }
> **기술요소**: 리버스 프록시, 로드밸런싱
> - (리버스 프록시) `서버 숨겨주는 대리자 역할`
> - (로드밸런싱) `부하 분산`

#### 가. 리버스 프록시와 로드밸런싱 개념도

```
┌─────────────────────────────────────────────────────────────────────┐
│              무중단 배포 기술요소: 리버스 프록시 vs 로드밸런싱          │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│    ┌──────────────────────────┐    ┌──────────────────────────┐     │
│    │      리버스 프록시         │    │       로드 밸런싱         │     │

│    │                          │    │                          │     │
│    │   ┌───┐     - Internal - │    │   ┌───┐                  │     │
│    │   │   │    ┌───┐ ┌───┐  │    │   │   │   ┌───┐  60%     │     │
│    │   │ 👤│───►│리버│ │███│  │    │   │ 👤│──►│로드│──────►   │     │

│    │            │시  │ └───┘  │    │           │    │  └───┘  │     │

│    │                         │    │                  └───┘   │     │

```

#### 나. 리버스 프록시와 로드밸런싱 비교

| 구분 | 리버스 프록시 | 로드밸런싱 |
|:-----|:------------|:---------|
| **역할** | 서버 숨겨주는 대리자 역할 | 부하 분산 |
| **기능** | 클라이언트 요청을 적절한 서버로 전달 | 서버 간 트래픽을 효율적으로 분산 |
| **장점** | 보안 강화, 캐싱, SSL 터미네이션 | 가용성 향상, 확장성 확보 |
| **도구 예시** | Nginx, Apache, HAProxy | AWS ELB, Nginx, F5 |

- 리버스 프록시는 클라이언트 요청을 적절한 서버로 전달하고, 로드밸런싱은 서버 간 트래픽을 효율적으로 분산시켜 **시스템의 무중단 배포의 핵심 역할 수행**

### ⭐ 차별점 키워드

| 배포 방식 | 핵심 차별점 |
|:---------|:----------|
| **롤링 배포** | 점진적 교체, 추가 인프라 불필요, 두 버전 혼재 가능 |
| **블루-그린 배포** | 즉시 전환/롤백, 2배 인프라 필요, 사전 테스트 가능 |
| **카나리 배포** | 단계적 트래픽 전환, 실시간 모니터링, 리스크 최소화 |
| **핵심 기술** | 리버스 프록시(대리자) + 로드밸런싱(부하분산) |

### 📈 상위 토픽 계층도

