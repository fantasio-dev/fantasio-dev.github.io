### 🔑 핵심 키워드 3개

| 키워드 | 설명 |
|:------|:-----|
| **추적성** | 요구사항이 어디서 왔고, 어디로 연결되는지 따라가는 것 |
| **변경관리** | 요구사항이 바뀌면 관련된 설계/코드/테스트를 찾아서 수정 |
| **누락방지** | 빠진 요구사항 없이 모든 것이 구현되었는지 확인 |

---

### 📖 등장배경

```
🌐 과거: 대규모 프로젝트에서 요구사항이 수백 개씩 쏟아짐

💡 문제: "이 요구사항 어디에 반영됐지? 변경되면 뭐 수정해야 하지?"

✅ 해결: 요구사항 추적표로 연결관계 명시!
   → 요구사항 ↔ 설계 ↔ 코드 ↔ 테스트 연결
   → 변경 시 영향도 즉시 파악
   → 누락 없는 개발/검증 보장
```

---

### 📝 개념 정의

> **요구사항 추적표(RTM)**: 요구사항과 개발산출물 간 연결관계를 관리하는 문서

| 구분 | 정의 (22자 이내) |
|:-----|:----------------|
| **전방 추적** | 요구사항 → 설계 → 코드 추적 |
| **후방 추적** | 테스트 → 코드 → 요구사항 역추적 |

### 쉬운 비유

| 요구사항 추적표 | 택배 추적 시스템 |
|:-----|:-----|
| 요구사항 ID로 추적 | 송장번호로 배송 현황 추적 |
| 어디까지 구현됐나 확인 | 지금 어디까지 왔나 확인 |

---

### 🏗️ 기술요소

#### 그룹 1: 식별 항목 `아설유우`

| 항목 | 설명 |
|:-----|:-----|
| **아**이디(ID) | 각 요구사항을 식별하는 고유 번호 |
| **설**명 | 요구사항의 기능적/비기능적 내용 |
| **유**형 | 기능(FR) 또는 비기능(NFR) 구분 |
| **우**선순위 | High/Medium/Low 중요도 |

#### 그룹 2: 추적 항목 `연설코테`

| 항목 | 설명 |
|:-----|:-----|
| **연**관 모듈 | 요구사항이 적용될 시스템/모듈 |
| **설**계 참조 | 반영된 설계 문서 ID |
| **코**드 참조 | 구현된 코드 파일/클래스 |
| **테**스트 케이스 | 검증용 테스트 케이스 ID |

## 🎯 기술사 수준 설명

{: .highlight }
> **요구사항 추적표**: 변경영향분석 → ID기반 추적, 설계연결, 테스트매핑 → 요구사항 연결관계 관리문서
> - (식별) `아설유우`
> - (추적) `연설코테`
> - ⭐ **차별점**: Bidirectional Traceability - 전방/후방 양방향 추적으로 완전한 영향분석 가능

---

### 📝 개념 정의

#### 🔑 핵심 키워드 3개

| 키워드 | 설명 |
|:------|:-----|
| **추적성(Traceability)** | 요구사항이 개발 산출물로 연결되는 관계 |
| **변경영향분석** | 요구사항 변경 시 영향받는 산출물 식별 |
| **커버리지 확인** | 모든 요구사항이 구현/테스트됨을 검증 |

---

#### 📖 등장배경

```
🌐 과거: 대규모 시스템 개발에서 요구사항과 산출물 간 연결 파악 어려움

💡 문제: "요구사항 변경 시 어떤 설계/코드/테스트를 수정해야 하는가?"

✅ 해결: RTM으로 요구사항-산출물 간 연결관계 체계적 관리
   → 요구사항 ID 기반 전방/후방 추적
   → 변경영향범위 즉시 파악
   → 누락 없는 검증 보장
```

---

#### 📝 정의

| 구분 | 정의 |
|:-----|:-----|
| **정의** | 시스템 개발 과정에서 요구사항 변경에 따라 시스템 영향을 프로젝트 초기부터 추적할 수 있도록 만든 요구사항 설명 문서 |
| **목적** | 요구사항과 설계/코드/테스트 간 연결관계를 명시하여 변경영향 분석 및 누락 방지 |

### 🏗️ 구성요소

#### 구성도

```
┌──────────────────────────────────────────────────────────────────────────────────────────┐
│                           요구사항 추적표 (RTM)                                            │
├──────────┬───────────────┬──────────┬──────────┬───────────┬───────────┬────────────────┤
│ 요구사항 ID│   요구사항 설명  │   유형   │  우선순위  │ 연관 모듈  │ 설계 참조 │ 테스트케이스 ID │
├──────────┼───────────────┼──────────┼──────────┼───────────┼───────────┼────────────────┤
│  RQ-001  │ 로그인 기능      │   FR     │   High   │ AuthModule│  DS-001   │    TC-001      │
│  RQ-002  │ 비밀번호 암호화   │   NFR    │   High   │ AuthModule│  DS-002   │    TC-002      │
│  RQ-003  │ 응답시간 3초이내  │   NFR    │  Medium  │   전체     │  DS-003   │    TC-003      │
└──────────┴───────────────┴──────────┴──────────┴───────────┴───────────┴────────────────┘
```

- 요구사항명에 따라 요구사항 항목을 구분하여 변경사항을 추적함

---

#### 그룹 1: 식별 항목 `아설유우`

| 항목 | 설명 | 예시 |
|:-----|:-----|:-----|
| **아**이디(ID) | 각 요구사항을 식별할 수 있는 고유 ID | RQ-001, RQ-002 |
| **설**명 | 해당 요구사항의 기능적 또는 비기능적 설명 | 사용자는 아이디와 비밀번호를 입력하여 로그인할 수 있어야 함 |
| **유**형 | 기능적 요구사항(FR) 또는 비기능적 요구사항(NFR) | 기능적 요구사항 (FR) |
| **우**선순위 | 요구사항 중요도 및 긴급도 | High / Medium / Low |

---

#### 그룹 2: 추적 항목 `연설코테`

| 항목 | 설명 | 예시 |
|:-----|:-----|:-----|
| **연**관 시스템/모듈 | 요구사항이 적용될 시스템 또는 모듈 | 사용자 인증 모듈 (AuthModule) |
| **설**계 문서 참조 | 해당 요구사항이 반영된 설계 문서 링크 또는 참조 ID | 설계 문서 ID: DS-001 |
| **코**드 참조 | 구현된 코드의 파일명 또는 클래스, 함수 | AuthModule.java |
| **테**스트 케이스 ID | 요구사항 검증을 위한 테스트 케이스 ID | TC-001 |

---

### 🔄 활용

| 활용 | 설명 |
|:-----|:-----|
| **요구사항 검증** | 기능 요구사항 RQ-001이 설계 문서 DS-001과 코드 AuthModule.java에 반영되었는지 검토 |
| **테스트 커버리지 확인** | RQ-002와 연결된 테스트 케이스 TC-002의 실행 여부 확인 |
| **변경 관리** | RQ-003의 변경으로 인해 DS-003 문서와 AuthAPI.java 수정 필요 여부 파악 |
| **품질 보증** | RQ-005 관련 테스트 TC-005가 Fail되었으며, 원인 분석 후 코드 수정 필요 |
| **프로젝트 진행 관리** | 현재 RQ-004의 개발이 완료되었으며, 테스트 대기 중 |
| **규정 및 법적 준수 확인** | GDPR 관련 요구사항 RQ-010이 개발 및 테스트 완료되었는지 점검 |
| **결함 추적 및 개선** | RQ-007에서 발견된 결함이 수정되었는지 확인 및 테스트 완료 |

- 다양한 관점에서 요구사항 추적표 활용으로 시스템 구축 품질 확보
- 요구사항 명세서를 기반으로 사업 초기에 작성 후 변경사항 지속적으로 업데이트
