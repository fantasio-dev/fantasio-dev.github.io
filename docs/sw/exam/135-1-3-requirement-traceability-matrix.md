## 고딩수준 요약

<details markdown="block">
<summary style="cursor: pointer; font-weight: bold; color: #1e88e5;">📚 고딩수준으로 이해하기 (펼치기)</summary>

### 요구사항 추적표가 뭐야?

**요구사항 추적표(RTM)**는 쉽게 말해서 **"요구사항 체크리스트 + 연결고리 표"**예요!

학교에서 과제할 때 생각해보세요:
- 선생님이 "이것저것 해오세요" 라고 했는데
- 뭘 했고, 뭘 안 했는지 헷갈릴 때 있죠?
- 그래서 **체크리스트**를 만들어서 확인하잖아요!

```
📋 과제 체크리스트
☑️ 국어 독후감 → 제출함 → 선생님 확인 완료
☑️ 수학 문제집 → 제출함 → 오답 3개
☐ 영어 단어 → 아직 안함
```

요구사항 추적표도 똑같아요!

```
📋 시스템 요구사항 추적표
☑️ 로그인 기능 → 설계완료 → 코드작성 → 테스트통과 ✅
☑️ 회원가입 → 설계완료 → 코드작성 → 테스트중
☐ 비밀번호 찾기 → 설계중
```

### 왜 필요해?

| 이유 | 예시 |
|:---|:---|
| **빠진 거 없는지 확인** | "로그인 기능 구현했나?" → 표 보면 바로 알 수 있어! |
| **변경사항 추적** | "어제 요구사항 바뀌었는데 어디 고쳐야 해?" → 표 보면 연결된 것들 다 보여! |
| **테스트 확인** | "이 기능 테스트 했나?" → 표에서 바로 확인! |

### 암기팁 🎯

**"요구사항 추적표 = 요구사항 GPS! 어디서 왔고, 어디로 갔는지 다 추적!"**

## 기술사수준 답안

### 1. 요구사항 변경 확인, 요구사항 추적표 정의

| 구분 | 내용 |
|:---|:---|
| **정의** | 시스템 개발 과정에서 요구사항 변경에 따라 시스템 영향을 프로젝트 초기부터 추적할 수 있도록 만든 요구사항 설명 문서 |
| **목적** | 요구사항과 설계/코드/테스트 간 연결관계를 명시하여 변경영향 분석 및 누락 방지 |

---

### 2. 요구사항 추적표 구성도 및 주요 항목

#### 구성도

```
┌──────────────────────────────────────────────────────────────────────────────────────────┐
│                           요구사항 추적표 (RTM)                                            │
├──────────┬───────────────┬──────────┬──────────┬───────────┬───────────┬────────────────┤
│ 요구사항 ID│   요구사항 설명  │   유형   │  우선순위  │ 연관 모듈  │ 설계 참조 │ 테스트케이스 ID │
├──────────┼───────────────┼──────────┼──────────┼───────────┼───────────┼────────────────┤
│  RQ-001  │ 로그인 기능      │   FR     │   High   │ AuthModule│  DS-001   │    TC-001      │
│  RQ-002  │ 비밀번호 암호화   │   NFR    │   High   │ AuthModule│  DS-002   │    TC-002      │
│  RQ-003  │ 응답시간 3초이내  │   NFR    │  Medium  │   전체     │  DS-003   │    TC-003      │
└──────────┴───────────────┴──────────┴──────────┴───────────┴───────────┴────────────────┘
```

- 요구사항명에 따라 요구사항 항목을 구분하여 변경사항을 추적함

---

#### 식별 항목 — (식별) `아설유우`

| 항목 | 설명 | 예시 |
|:---|:---|:---|
| **요구사항 ID (아이디)** | 각 요구사항을 식별할 수 있는 고유 ID | RQ-001, RQ-002 |
| **요구사항 설명** | 해당 요구사항의 기능적 또는 비기능적 설명 | 사용자는 아이디와 비밀번호를 입력하여 로그인할 수 있어야 함 |
| **요구사항 유형** | 기능적 요구사항(FR) 또는 비기능적 요구사항(NFR) | 기능적 요구사항 (FR) |
| **우선순위** | 요구사항 중요도 및 긴급도 | High / Medium / Low |

---

#### 추적 항목 — (추적) `연설코테`

| 항목 | 설명 | 예시 |
|:---|:---|:---|
| **연관 시스템/모듈** | 요구사항이 적용될 시스템 또는 모듈 | 사용자 인증 모듈 (AuthModule) |
| **설계 문서 참조** | 해당 요구사항이 반영된 설계 문서 링크 또는 참조 ID | 설계 문서 ID: DS-001 |
| **코드 참조** | 구현된 코드의 파일명 또는 클래스, 함수 | AuthModule.java |
| **테스트 케이스 ID** | 요구사항 검증을 위한 테스트 케이스 ID | TC-001 |

---

### 3. 요구사항 추적표 활용

| 활용 | 설명 |
|:---|:---|
| **요구사항 검증** | 기능 요구사항 RQ-001이 설계 문서 DS-001과 코드 AuthModule.java에 반영되었는지 검토 |
| **테스트 커버리지 확인** | RQ-002와 연결된 테스트 케이스 TC-002의 실행 여부 확인 |
| **변경 관리** | RQ-003의 변경으로 인해 DS-003 문서와 AuthAPI.java 수정 필요 여부 파악 |
| **품질 보증** | RQ-005 관련 테스트 TC-005가 Fail되었으며, 원인 분석 후 코드 수정 필요 |
| **프로젝트 진행 관리** | 현재 RQ-004의 개발이 완료되었으며, 테스트 대기 중 |
| **규정 및 법적 준수 확인** | GDPR 관련 요구사항 RQ-010이 개발 및 테스트 완료되었는지 점검 |
| **결함 추적 및 개선** | RQ-007에서 발견된 결함이 수정되었는지 확인 및 테스트 완료 |

- 다양한 관점에서 요구사항 추적표 활용으로 시스템 구축 품질 확보
- 요구사항 명세서를 기반으로 사업 초기에 작성 후 변경사항 지속적으로 업데이트
