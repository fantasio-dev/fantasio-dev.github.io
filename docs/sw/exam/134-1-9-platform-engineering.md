### 🔑 핵심 키워드 3개

| 키워드 | 설명 |
|:------|:-----|
| **개발자 플랫폼** | 개발자가 코드 작성에만 집중할 수 있도록 인프라를 자동화해주는 내부 플랫폼 |
| **셀프서비스** | 개발자가 직접 필요한 인프라를 버튼 클릭으로 생성할 수 있는 환경 |
| **DevOps 진화** | DevOps의 다음 단계로, 개발자 생산성 향상에 집중 |

---

### 📖 등장배경

```
🌐 과거: DevOps로 개발과 운영이 합쳐졌지만, 개발자가 알아야 할 도구가 너무 많아짐

💡 문제: "개발자가 코드도 짜고, Kubernetes도 알고, Terraform도 알아야 한다고?"

✅ 해결: 플랫폼 엔지니어링 팀이 복잡한 인프라를 추상화해서 제공!
   → 개발자는 "Deploy" 버튼만 누르면 됨
   → 복잡한 설정은 플랫폼 팀이 뒤에서 처리
   → 개발 속도 UP, 인지 부하 DOWN
```

---

### 📝 개념 정의

> **플랫폼 엔지니어링**: 개발자 생산성 향상을 위한 내부 플랫폼 구축 엔지니어링

| 개념 | 비유 |
|:-----|:-----|
| **플랫폼 엔지니어링** | 뷔페 레스토랑 - 요리사(플랫폼팀)가 음식을 차려놓으면 손님(개발자)은 원하는 것만 가져가면 됨 |
| **DevOps** | 음식 재료와 조리법을 주고 "알아서 요리해"라고 하는 것 |

---

### 🏗️ 기술요소

#### 핵심 역할 `인분확모`

| 역할 | 설명 |
|:-----|:-----|
| **인**프라 자동화 | Terraform, Ansible로 인프라를 코드로 관리 |
| **분**산 시스템 설계 | MSA, 마이크로서비스 아키텍처 지원 |
| **확**장성 최적화 | 캐싱, 로드밸런싱, 샤딩 등 성능 최적화 |
| **모**니터링/로깅 | 실시간 모니터링, 경고 시스템 구축 |

#### 기술 스택 `I-T-K-C`

| 기술 | 설명 |
|:-----|:-----|
| **I**aC | Infrastructure as Code - 인프라를 코드로 관리 |
| **T**erraform | 클라우드 인프라 프로비저닝 도구 |
| **K**8s | Kubernetes 컨테이너 오케스트레이션 |
| **C**I/CD | 지속적 통합/배포 파이프라인 |

---

### ⭐ 차별점 키워드

**"개발자 경험(DX) 중심"** - DevOps가 "어떻게 배포할까?"라면, 플랫폼 엔지니어링은 "개발자가 얼마나 쉽게 배포할 수 있을까?"에 집중

---

### 📋 6하원칙 요약

| 구분 | 내용 |
|:-----|:-----|
| **누가(Who)** | 플랫폼 엔지니어링 팀 (Cross Functional Team) |
| **무엇을(What)** | 내부 개발자 플랫폼(IDP) 구축 |
| **언제(When)** | DevOps 도입 후 복잡성이 증가했을 때 |
| **어디서(Where)** | 클라우드 네이티브 환경, 대규모 조직 |
| **왜(Why)** | 개발자 생산성 향상, 인지 부하 감소 |
| **어떻게(How)** | 셀프서비스 포털, 자동화 도구, 추상화 계층 제공 |

## 🎯 기술사 수준 설명

{: .highlight }
> **플랫폼 엔지니어링**: DevOps 복잡성 증가 → 셀프서비스, IDP, Cross Functional Team → 개발자 생산성 향상을 위한 내부 플랫폼 구축 엔지니어링
> - (역할) `인분확모보`
> - (기술) `I-T-K-C`
> - ⭐ **차별점**: Developer Experience(DX) 중심 - 개발자가 인프라 복잡성 없이 비즈니스 로직에만 집중

---

### 🔑 핵심 키워드 3개

| 키워드 | 설명 |
|:------|:-----|
| **IDP** | Internal Developer Platform - 내부 개발자 플랫폼 |
| **셀프서비스** | 개발자가 직접 인프라 리소스를 프로비저닝 |
| **Cross Functional Team** | 다기능 팀으로 계층 간 커뮤니케이션 조정 역할 |

---

### 📖 등장배경

```
🌐 과거: DevOps 도입으로 개발-운영 경계가 허물어짐

💡 문제: "소프트웨어 복잡성 증가로 개발자 인지 부하 급증"

✅ 해결: 플랫폼 엔지니어링으로 추상화 계층 제공!
   → 일관된 CI/CD 파이프라인 유지
   → 최소 오버헤드로 가치 있는 소프트웨어 생산
   → Time to Market 단축
```

| 등장 배경 | 설명 |
|:---------|:-----|
| **소프트웨어 복잡성 증가** | 빠른 배포 필요하지만 수행 과정에서의 마찰 감소 필요 |
| **개발 생산성 향상** | 최소 오버헤드로 가치 있는 소프트웨어 생산 지원 |
| **자동화된 워크로드** | 리포지토리부터 생산까지 CI/CD 파이프라인 일관성 유지 |

---

### 📝 개념 정의

> **플랫폼 엔지니어링**: 여러 애플리케이션과 서비스를 지원하고 수명 주기를 관리하기 위해 소프트웨어 플랫폼을 설계하고 구축하여 내부 사용자에게 제공하는 엔지니어링

### 🏗️ 구성요소

#### 그룹 1: 핵심 역할 `인분확모`

| 항목 | 설명 | 예시 |
|:-----|:-----|:-----|
| **인**프라 자동화 | IaC 적용하여 인프라 관리 자동화 | Terraform, Ansible |
| **분**산 시스템 설계 | MSA, 분산 시스템 지원 플랫폼 설계 | Service Mesh, API Gateway |
| **확**장성/성능 최적화 | 성능 최적화 기술 이해 및 설계 반영 | Caching, Load balancing, Sharding |
| **모**니터링/로깅 | 실시간 모니터링 및 디버깅 환경 구축 | Prometheus, Grafana, ELK Stack |

#### 그룹 2: 핵심 기술 스택 `I-T-K-C`

| 항목 | 설명 | 예시 |
|:-----|:-----|:-----|
| **I**aC | Infrastructure as Code - 인프라 코드화 | Terraform, Pulumi, CDK |
| **T**ooling | 자동화 도구 및 구성 관리 | Ansible, Chef, Puppet |
| **K**8s/Container | 컨테이너 오케스트레이션 | Kubernetes, Docker, Helm |
| **C**I/CD Pipeline | 지속적 통합/배포 파이프라인 | Jenkins, GitLab CI, ArgoCD |

### 📊 DevOps vs Platform Engineering vs SRE 비교

| 구분 | DevOps | Platform Engineering | SRE |
|:-----|:-------|:--------------------|:----|
| **핵심 목적** | 개발-운영 통합 | 개발자 생산성 향상 | 서비스 안정성 확보 |
| **핵심 지표** | 배포 빈도, MTTR | 개발자 만족도, Time to Market | SLO, SLA, 오류 예산 |
| **접근 방식** | 문화 + 도구 | 내부 플랫폼 제품화 | 소프트웨어 엔지니어링 원칙 적용 |
| **주요 활동** | CI/CD, 협업 강화 | IDP 구축, 셀프서비스 제공 | 온콜, 장애 대응, 자동화 |
| **팀 구조** | 기능별 분산 | Cross Functional Team | 전담 SRE 팀 |

### ⭐ 차별점 키워드

| 차별점 | 설명 |
|:-------|:-----|
| **Developer Experience(DX) 중심** | 개발자가 인프라 복잡성 없이 비즈니스 로직에만 집중할 수 있도록 추상화 계층 제공 |
| **내부 제품(Internal Product)** | IDP를 하나의 제품으로 보고, 개발팀을 고객으로 대함 |
| **Golden Path** | 모범 사례를 템플릿화하여 개발자가 빠르게 시작할 수 있도록 지원 |
