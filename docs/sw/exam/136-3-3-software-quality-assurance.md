### 💡 한 줄 정의

> **소프트웨어 품질보증**: 소프트웨어 제품이 요구사항과 일치하는가를 확인하는 개발단계 전체에 걸친 체계적인 작업

### 🎭 쉬운 비유

| 구분 | 비유 |
|:-----|:-----|
| **품질보증 (QA)** | 요리 과정 전체를 감독하는 주방장 - "레시피대로 만들고 있나?" |
| **품질통제 (QC)** | 완성된 요리를 검사하는 맛 테스터 - "맛이 괜찮나?" |
| **인스펙션** | 레시피 자체를 미리 검토하는 회의 - "이 레시피에 문제없나?" |

### ❓ 왜 필요한가? (3줄 요약)

1. **SW 생산성 향상**: 체계적인 품질관리로 개발 효율 증대
2. **납기준수, 비용절감**: 결함 조기 발견으로 재작업 비용 감소
3. **제품의 견고성/확장성**: 품질 확보를 통한 제품 신뢰성 향상

## 🎯 기술사 수준 설명

{: .highlight }
> **소프트웨어 품질보증**: ISO 25010, QA/QC, Inspection → 개발단계 전체의 체계적 품질확보 활동
> - (1) `기신사성유보호안` (소프트웨어 품질의 의미)
> - (2) `사기관-품형리/품품품-테결검` (품질보증의 목적과 기능)
> - (3) `동사개검재후` (인스펙션과 인스펙션 프로세스)
> - ⭐ **차별점**: 인스펙션은 실행 없이 산출물을 정적 분석하는 공식적 검토 기법

### 1. 소프트웨어 품질의 의미

{: .highlight }
> **소프트웨어 품질**: ISO/IEC 25010, 품질특성 8가지, QA/QC
> - (품질특성) `기신사성유보호안`
> - (품질관리) `QA-QC`
> - (품질확보) `Big Q - Small Q`

#### 가. 소프트웨어 품질 개념도

```
┌─────────────────────────────────────────────────────────────────────┐
│                      소프트웨어 품질 개념도                           │
├────────────────────┬────────────────────┬───────────────────────────┤
│    품질특성 충족     │    품질관리 (QM)    │        품질확보           │
├────────────────────┼────────────────────┼───────────────────────────┤
│                    │                    │                          │
│  ┌──────────────┐  │  ┌──────┐ ┌──────┐ │  ┌────────┐  ┌────────┐  │
│  │ ISO/IEC 25010│  │  │품질보증│ │품질통제│ │  │ 고객만족 │  │ BigQ   │  │
│  └──────────────┘  │  │  QA  │ │  QC  │ │  └────────┘  └────────┘  │
│                    │  └──────┘ └──────┘ │                          │
│  • 기능적합성       │                    │  ┌────────┐  ┌────────┐  │
│  • 신뢰성          │  Inspection        │  │ 무결함  │  │ SmallQ │  │
│  • 사용성          │  Audit             │  │   SW   │  └────────┘  │
│  • 성능효율성       │  Review            │  └────────┘              │
│  • 유지보수성       │                    │                          │
│  • 보안성          │  ParetoChart       │                          │
│  • 호환성          │  Fishbone          │                          │
│  • 안전성          │  PDPC              │                          │
│                    │                    │                          │
└────────────────────┴────────────────────┴───────────────────────────┘
```

#### 나. 소프트웨어 품질의 정의

| 구분 | 정의 |
|:-----|:-----|
| **사전적 의미** | 소프트웨어가 정의된 요구사항을 정확하고 안정적으로 충족하며, 사용자의 기대 수준에 부합하는 기능적·비기능적 특성을 보장하는 정도 |
| **수학적 의미** | 기능 일치도 = 정상기능수행수/요구기능수, 결함률 = 1-(결함수/전체기능수), 신뢰성 = MTTF/(MTTF+MTTR) |

- 품질을 객관적으로 검증하기 위한 제3자 입장의 **품질보증(Quality Assurance)** 수행이 중요

#### 다. ISO/IEC 25010:2023 기반 소프트웨어 품질 특성 `기신사성유보호안`

| 주특성 | 부특성 | 설명 |
|:------|:------|:-----|
| **기**능적합성 (Functional Suitability) | 기능완전성, 기능정확성, 기능적절성 | 요구된 기능이 모두 구현되어 있는지, 정확하게 동작하는지 평가 |
| **신**뢰성 (Reliability) | 성숙성, 가용성, 결함허용성, 복구성 | 시스템의 오류 발생 빈도, 사용 가능할 확률, 결함 시 계속 동작할 수 있는 능력 평가 |
| **사**용성 (상호작용능력, Interaction Capability) | 적절 인지성, 학습성, 운용성, 사용자 오류 방지성, 사용자 참여도, 사용 포함성, 사용자 지원성, 자기 설명성 | 사용자가 시스템을 얼마나 쉽게 사용할 수 있는지 평가 |
| **성**능효율성 (Performance Efficiency) | 시간 반응성, 자원 효율성, 용량성 | 시스템 응답 시간, 자원 효율성, 최대 용량 평가 |
| **유**지보수성 (Maintainability) | 모듈성, 재사용성, 분석성, 수정가능성, 시험가능성 | 시스템의 변경이 얼마나 용이한지 평가 |
| **보**안성 (Security) | 기밀성, 무결성, 부인방지, 책임성, 인증성, 저항성 | 데이터 보호, 접근 제어, 사용자 인증 능력 평가 |
| **호**환성 (Compatibility) | 공존성, 상호운용성 | 다른 시스템과의 통합 및 정보 교환 가능성 평가 |
| **안**전성 (Safety) | 운영 제약, 위험 식별, 실패 안전, 위험 경고 | 시스템이 안전하게 작동하고 위험을 방지하는 능력 평가 |
| 유연성 (Flexibility) | 적응성, 확장성, 설치성, 대체성 | 다른 환경에 쉽게 적응하고 확장할 수 있는지 평가 |

### 2. 소프트웨어 품질보증의 목적과 기능

{: .highlight }
> **품질보증의 목적과 기능**: 고객만족, 품질향상, 기술/관리/운영 관점
> - (목적-고객) `요신서`
> - (목적-품질) `표오지복`
> - (기능-기술) `품형리`
> - (기능-관리) `품품품`
> - (기능-운영) `테결검`

#### 가. 소프트웨어 품질보증의 목적

| 관점 | 목적 | 설명 |
|:-----|:-----|:-----|
| **고객만족 측면** | **요**구사항 만족 | 사용자의 기대와 요구를 충족하는 결과물 제공 |
| | **신**뢰성 확보 | 안정적이고 예측 가능한 시스템 동작 보장 |
| | **서**비스 품질 보장 | 사용성과 보안성을 갖춘 사용자 중심 품질 확보 |
| **품질향상 측면** | **표**준 준수 및 절차화 | 개발 프로세스의 일관성과 체계성 확보 |
| | **오**류 예방 및 리스크 완화 | 시스템이 요구된 시간 동안 사용 가능할 확률 평가 |
| | **지**속적 개선 | 개발 초기 단계부터 결함 방지를 위한 관리 체계 적용 |
| | **복**구성 | 품질 측정과 리뷰를 통한 프로세스 및 산출물 개선 |

- 목적으로 방향성 제시, 이를 실현하기 위한 실행 수단과 활동 체계를 제공하는 기능 정의가 요구

#### 나. 소프트웨어 품질보증의 기능

| 관점 | 기능 | 설명 |
|:-----|:-----|:-----|
| **기술적 관점** | **품**질 측정 및 메트릭 관리 | 품질 속성(결함 수, 커버리지 등)에 대한 정량적 데이터를 수집하고 분석함 |
| | **형**상 관리 지원 | 소프트웨어 산출물의 버전과 변경사항을 체계적으로 추적 및 관리함 |
| | **리**스크 분석 / 예방적 품질 확보 | 잠재적 결함 요인을 식별하고 사전 대응함으로써 문제 발생 가능성을 줄임 |
| **관리적 관점** | **품**질 계획 수립 | 품질 목표, 기준, 절차, 책임을 정의하고 이를 문서화하여 실행 기반을 마련함 |
| | **품**질 정책 및 표준 관리 | 조직 수준에서 품질 관련 정책, 표준, 가이드라인을 수립하고 일관되게 적용함 |
| | **품**질 감사 및 성과 평가 | 품질 활동의 실행 여부, 효과성, 문제점을 점검하고 개선 방안을 도출함 |
| **운영/실행 관점** | **테**스트 계획 및 수행 | 동적 검증을 통해 기능, 성능, 보안 등 제품의 품질을 실제로 확인함 |
| | **결**함 추적 및 품질 개선 활동 | 발견된 결함의 원인을 분석하고 지속적으로 품질을 개선하는 활동을 수행함 |
| | **검**토 및 점검 활동 | 요구사항, 설계, 코드 등의 산출물에 대해 정적 검토를 수행하여 오류를 조기에 발견함 |

- 품질보증 위해 Review(Management, Technical), Walkthrough, Inspection, Audit 등 주요 기법 이해 요구
- 개발 초기에 결함을 조기에 발견하여 품질 저하와 재작업 비용을 줄이기 위해 **인스펙션(Inspection)** 수행

### 3. 인스펙션(Inspection)과 인스펙션 프로세스(Inspection Process)

{: .highlight }
> **인스펙션**: 정적 분석, 공식적 절차, 결함 조기 발견
> - (특징) `정공초`
> - (역할) `검작기진`
> - (프로세스) `동사개검재후`

#### 가. 인스펙션(Inspection) 개요

| 구분 | 내용 |
|:-----|:-----|
| **개념** | 요구사항, 설계서, 코드 등 소프트웨어 산출물을 **공식적인 절차**에 따라 정적 분석(실행 없이)하고, **결함을 조기에 식별**하기 위한 품질보증 기법 |
| **특징 1** | **정**적(Static) 검토 기법 - 소프트웨어를 실행하지 않고 문서나 코드 자체를 분석하여 결함을 찾음 |
| **특징 2** | **공**식적이고 구조화된 절차 - 사전 준비, 검토 회의, 결함 기록, 후속 조치 등 명확한 절차와 역할 분담이 이루어짐 |
| **특징 3** | **초**기 결함 조기 발견 - 개발 초기 단계에서 오류를 식별할 수 있어, 수정 비용과 개발 지연을 최소화할 수 있음 |

- 검토자(Reviewer), 작성자(Author), 기록자(Recorder), 진행자(Moderator) 등이 역할을 나누어 수행하며, **형식적 회의와 체크리스트 기반**으로 검토

#### 나. 인스펙션 프로세스(Inspection Process) `동사개검재후`

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ Planning │───►│ Overview │───►│Preparation│───►│ Meeting  │───►│  Rework  │───►│Follow-up │
│  (계획)   │    │ (사전설명) │    │  (준비)   │    │  (회의)   │    │  (재작업) │    │ (후속조치) │
└──────────┘    └──────────┘    └──────────┘    └──────────┘    └──────────┘    └──────────┘
     │               │               │               │               │               │
     ▼               ▼               ▼               ▼               ▼               ▼
  대상 선정        기준 공유       개별 사전 검토    결함 발표/기록    결함 수정/반영   수정 확인/재검토
  참여자 지정      산출물 설명     체크리스트 활용   결함 식별/정리     재검토 준비      종료 판단
```

| 프로세스 | 주요활동 | 설명 |
|:--------|:--------|:-----|
| ① **동**료검토 계획수립 (Planning) | 검토 범위 및 역할 정의 | 인스펙션 대상 산출물 선정, 검토 일정 수립, 참여자 역할 분담 |
| ② **사**전 설명회 (Overview) | 인스펙션 기준 및 산출물 공유 | 검토 대상의 목적, 구조, 사용 기술 등을 팀원에게 설명하고 체크리스트 및 기준 공유 |
| ③ **개**별검토 (Preparation) | 독립적 결함 식별 | 각 참여자가 문서를 개인적으로 분석하며 결함 후보를 식별 및 기록 |
| ④ **검**토회의 실시 (Meeting) | 결함 수집 및 기록 | 판독자가 문서를 설명하고, 팀원들은 식별된 결함을 공유하며 리더가 이를 기록 |
| ⑤ **재**작업 (Rework) | 결함 수정 | 저자가 검토 회의에서 식별된 결함을 수정하고 문서나 코드에 반영 |
| ⑥ **후**속조치 (Follow-up) | 결과 검토 및 품질 데이터 관리 | 재검토를 통해 수정 내역 확인, 검토 품질 통계 수집, 지속적 개선 활동 반영 |

- 형식적인 회의보다 **결함 발견의 실효성**과 **피드백 품질 확보**에 중점을 두며 효과적 수행이 필요
- 인스펙션 프로세스는 산출물에 내재된 결함을 **구조화된 절차**를 통해 조기에 식별하고 수정함으로써 품질을 향상시키는 활동

---

### ⭐ 차별점 키워드

| 구분 | 차별점 |
|:-----|:------|
| **QA vs QC** | QA는 프로세스 중심(예방), QC는 제품 중심(검출) |
| **인스펙션 vs 워크스루** | 인스펙션은 공식적/체계적, 워크스루는 비공식적/저자 주도 |
| **정적 검토 vs 동적 테스트** | 정적은 실행 없이 검토, 동적은 실행하며 검증 |
| **ISO 25010:2023** | 8가지 주특성 (기능적합성, 신뢰성, 사용성, 성능효율성, 유지보수성, 보안성, 호환성, 안전성) + 유연성 |

### 📈 상위 토픽 계층도

```
소프트웨어 공학
├── 품질 관리 (Quality Management)
│   ├── 품질보증 (QA) ← 현재 토픽
│   │   ├── 인스펙션 (Inspection) ← 현재 토픽
│   │   ├── 워크스루 (Walkthrough)
│   │   ├── 감사 (Audit)
│   │   └── 리뷰 (Review)
│   ├── 품질통제 (QC)
│   │   └── 테스트 (Testing)
│   └── 품질 표준
│       ├── ISO/IEC 25010 ← 현재 토픽
│       ├── ISO 9001
│       └── CMMI
└── 프로젝트 관리
    └── 형상 관리
```
