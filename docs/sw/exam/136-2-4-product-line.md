### 한 줄 정의

> **제품계열(Product Line)**: 도메인 기반 핵심자산(Core Asset)을 개발하여 재사용성과 생산성을 극대화하는 SW 개발 방법론

### 쉬운 비유

| 일반 개발 | 제품계열 개발 |
|:---------|:------------|
| 🏠 매번 집을 처음부터 설계하고 짓기 | 🏘️ 기본 설계도(Core Asset)를 만들고, 고객 요구에 따라 변형하여 여러 집 짓기 |

### 왜 필요한가? (3줄 요약)

1. **비용 절감**: 공통 컴포넌트를 재사용하여 개발 비용 대폭 감소
2. **품질 향상**: 검증된 핵심 자산 사용으로 안정성과 신뢰성 확보
3. **시간 단축**: 유사 제품 개발 시 Time-to-Market 크게 단축

## 🎯 기술사 수준 설명

{: .highlight }
> **제품계열(Product Line)**: Core Asset, 재사용성, 생산성 → 도메인 기반 SW 개발 방법론
> - (1) `핵품생초` (개념과 특징)
> - (2) `도응관` (활용 기술)
> - (3) `비기조투` (고려사항)
> - ⭐ **차별점**: Payoff Point 이후 누적 비용 역전 - 제품 수 증가할수록 ROI 급증

### 1. 제품계열(Product Line)의 개념과 특징

{: .highlight }
> **개념과 특징**: Core Asset, Domain Engineering, 재사용성
> - (정의) `핵심자산 기반 재사용`
> - (특징) `핵품생초`
> - (표준) `ISO/IEC 26550`

#### 가. 제품계열(Product Line) 방법론의 개념

```
┌─────────────────────────────────────────────────────────────────────┐
│                   Economics of Product Lines                        │
│                                                                     │

│         │                    ／                                     │
│         │                  ／                                       │
```

| 구분 | 정의 |
|:-----|:-----|
| **제품계열** | 도메인 기반 핵심자산(Core Asset) 개발로 재사용성·생산성 극대화 개발 방법론 |
| **핵심자산** | 제품군 공통으로 활용되는 아키텍처, 컴포넌트, 설계 산출물의 집합 |
| **관련표준** | ISO/IEC 26550 - Reference model for product line engineering and management |

#### 나. 제품계열(Product Line)의 특징 `핵품생초`

| 구분 | 특징 | 설명 |
|:-----|:-----|:-----|
| **기술 측면** | **핵**심 자산 기반 개발 | 공통 컴포넌트와 기능의 반복적 활용으로 개발 효율성 확보 |
|  | **품**질 향상 | 검증된 핵심 자산 사용으로 전체 제품의 안정성과 신뢰성 향상 |
| **비즈니스 측면** | **생**산성 향상 | 유사 제품 반복 개발 시 투입 시간과 비용 절감 가능 |
|  | **초**기 투자 필요 | 아키텍처 및 공통 자산 구축에 초기 비용과 인력 투입 필요 |

#### 다. 제품계열 방법론의 개념도

```
┌──────────────────────────────────────────────────────────────────┐
│                                                                  │
│   ┌─────────────────┐      Plug & Play      ┌────────────────┐  │

│   │  Development    │                        │  Development   │  │
│   └────────┬────────┘                        └───────┬────────┘  │
│            │                                         │           │
│     Asset 관리 ◀──────Repository에 저장────▶ Process 관리, 적용 │
│            │                                         │           │
│            │        ┌────────────────┐               │           │
│            └───────▶│   Management   │◀──────────────┘           │
│                     └────────────────┘                           │
│                                                                  │
│   ├─────── Domain Eng. ───────┤├───── Application Eng. ─────┤   │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

| 구분 | 역할 | 설명 |
|:-----|:-----|:-----|
| **Domain Engineering** | 핵심 자산 개발 | 제품군 공통 아키텍처, 컴포넌트, 도메인 모델 개발 |
| **Application Engineering** | 제품 개발 | 핵심 자산을 활용하여 개별 제품 구성 및 맞춤화 |
| **Management** | 자산 관리 | 핵심 자산의 버전, 형상, 변경 이력 통합 관리 |

### 2. 제품계열(Product Line)의 활용 기술

{: .highlight }
> **활용 기술**: 도메인 공학, 응용 공학, 관리 및 지원
> - (도메인공학) `개특아C`
> - (응용공학) `구테`
> - (관리지원) `형조메`

#### 가. 활용 기술 개념도

```
┌─────────────────────────────────────────────────────────────────┐
│                    Product Line 활용 기술                        │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│         ┌──────────────┬──────────────┬──────────────┐         │
│         │              │              │              │         │
│         ▼              ▼              ▼              │         │
│   ┌──────────┐  ┌──────────┐  ┌──────────────┐      │         │
│   │ 도메인   │  │  응용    │  │  관리 및     │      │         │
│   │  공학    │  │  공학    │  │   지원       │      │         │
│   └────┬─────┘  └────┬─────┘  └──────┬───────┘      │         │
│        │             │               │              │         │
│   ┌────┴────────────┴───────────────┴────────────┐  │         │
│   │     형상관리, 조직/프로세스 개선 및 지원      │  │         │
│   └─────────────────────────────────────────────────┘         │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### 나. 도메인 공학(Domain Engineering) `개특아C`

| 활용기술 | 설명 | 예시 |
|:--------|:-----|:-----|
| **개**념 모델링 | 특정 도메인 내의 핵심 개념, 관계, 속성을 식별하고 모델링 | UML 다이어그램 |
| **특**성 모델링 (Feature Modeling) | 모든 제품이 가질 수 있는 기능/비기능적 특성들을 모델링 | Feature Model |
| 제품계열 **아**키텍처 설계 | 확장성, 유연성, 재사용성을 고려한 핵심 자산의 기반 아키텍처 설계 | Reference Architecture |
| **C**BD (Component-Based Dev.) | 재사용 가능한 독립적인 SW 컴포넌트를 설계·구현하여 핵심 자산으로 활용 | 컴포넌트 라이브러리 |

#### 다. 응용 공학(Application Engineering) `구테`

| 활용기술 | 설명 | 예시 |
|:--------|:-----|:-----|
| 제품 **구**성 (Product Configuration) | 자동화도구를 기반으로 한 제품의 구성 | 제품 구성 도구 |
| 제품 **테**스트 (Product Testing) | 핵심 자산에 대한 테스트와 생성된 각 파생 제품에 대한 테스트 전략 구분 | 테스트 자동화 |

#### 라. 관리 및 지원 기술 `형조메`

| 활용기술 | 설명 | 예시 |
|:--------|:-----|:-----|
| **형**상 관리 (CM) | 모든 산출물에 대한 버전을 체계적으로 관리하여 일관성과 추적성 유지 | Git, SVN |
| **조**직 및 프로세스 개선 | CMMI, SPICE 등을 활용하여 조직 구조, 역할, 개발 프로세스에 변화 대응 | 프로세스 성숙도 향상 |
| **메**타모델링 및 모델 변환 | 도메인 모델, 피처 모델 등 다양한 모델들을 정의하고, 모델 간의 변환 자동화·효율화 | MDA, MDD |

### 3. 제품계열(Product Line) 방법론의 고려사항

{: .highlight }
> **고려사항**: 비즈니스, 기술, 조직, 투자·위험
> - (관점) `비기조투`
> - (비즈니스) `범시`
> - (기술) `아핵가`
> - (조직관리) `역프인`
> - (투자위험) `ROI위`

#### 가. 고려사항 관점 분류 `비기조투`

| 관점 | 고려사항 | 설명 |
|:-----|:---------|:-----|
| **비**즈니스 관점 | 제품계열 **범**위 및 전략 수립 | 어떤 제품들을 제품계열로 묶을 것인지 정의 |
|  | **시**장 및 경쟁 환경 분석 | 제품계열이 목표 시장에서 경쟁 우위를 제공할 수 있는지 다각도 검토 |
| **기**술 관점 | **아**키텍처 설계 및 진화 | 공통 아키텍처 유연성 및 확장 가능성 검토, 미래 기술 변화에 대한 수용력 검토 |
|  | **핵**심 자산 식별 및 개발 고려 | 재사용 가능한 핵심 자산을 명확히 정의하고, 고품질로 개발할 수 있을지 고려 |
|  | **가**변성 관리 기법 및 도구 고려 | 제품 생성을 자동화할 수 있는 적절한 도구 검토 |
| **조**직 및 관리 관점 | 조직 구조 및 **역**할 정의 고려 | 핵심 자산 개발팀과 제품생산팀 간의 역할과 책임, 협업 및 의사소통 체계가 효과적으로 수립 |
|  | **프**로세스 및 거버넌스 수립 | 제품계열 전용 개발 프로세스 및 관리 절차가 정의되어 있는지 검토 |
|  | **인**력 및 문화적 변화 | 제품계열 전문 인력(도메인 전문가, 아키텍트)을 확보 |
| **투**자 및 위험 관리 관점 | **ROI** 분석 | 핵심 자산 구축에 필요한 초기 투자(시간, 비용) |
|  | **위**험 식별 및 완화 전략 | 기술적(아키텍처 실패), 조직적(저항), 시장적(수요 변화) 위험을 식별하고 완화 전략 수립 |

#### 나. 제품계열 적용 방법론 분류

| 구분 | 방법론 | 설명 |
|:-----|:-------|:-----|
| **선행적** (Proactive) | FORM | 초기 마케팅 및 제품 계획 단계부터 정해진 의사결정을 바탕으로 Feature 모델을 구성 |
| **후행적** (Reactive) | AHEAD | Algebraic Hierarchical Equation for Application Design, 반복적 Feature 정련(Refinement)을 통해 SW 제품계열을 점차 확대 |
|  | GP | Generative Programming, 상세한 Feature 모델을 바탕으로 프로그램 코드를 생성 |
| **추출식** (Extractive) | PuLSE | ProdUct Line Software Engineering, 기존 레거시 시스템에서 공통 자산을 추출하여 제품계열로 전환 |
