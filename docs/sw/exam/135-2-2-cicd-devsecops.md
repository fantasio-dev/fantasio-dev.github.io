### 한 줄 정의

> **CI/CD DevSecOps**: 지속적 통합/배포 파이프라인에 보안(Security)을 초기 단계부터 통합하는 개발 방법론

### 쉬운 비유

| DevOps | DevSecOps |
|:-------|:----------|
| 고속도로를 빠르게 달리는 자동차 | 안전벨트와 에어백을 갖춘 고속 자동차 |
| 개발 + 운영 협업 | 개발 + 보안 + 운영 협업 |

### 왜 필요한가? (3줄 요약)

1. **MSA/컨테이너 확산**: 마이크로서비스, 컨테이너 환경에서 보안 취약점 증가
2. **Shift-Left 보안**: 개발 초기 단계에서 보안 취약점 발견 시 수정 비용 절감
3. **자동화된 보안**: 수동 보안 검토의 한계 극복, CI/CD 속도 유지

### 1. 기업 관리 CI/CD 파이프라인의 필요성

{: .highlight }
> **CI/CD 필요성**: 컨테이너 기술 발전, MSA 확산, 테스트 중요성 대두
> - (배경) `컨MSA테` (컨테이너, MSA, 테스트)
> - (문제) 일반적인 배포 방식으로 관리 어려움
> - (해결) CI/CD 파이프라인 구성 필요

#### 가. CI/CD 파이프라인 필요성 배경

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    CI/CD 파이프라인 필요성                               │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  ┌──────────────────┐                      ┌──────────────────────────┐ │
│  │ 컨테이너 기술 발전 │                      │                          │ │
│  └──────────────────┘                      │                          │ │
│           │                                │      CI/CD               │ │
│           ▼              일반적인 배포       │      파이프라인          │ │
│  ┌──────────────────┐    방식으로          │      구성 필요           │ │
│  │   MSA 확산       │───▶ 관리 어려움 ─────▶│                          │ │

│           │                                │                          │ │
│           ▼                                │                          │ │

│                                                                         │

```

| 배경 요인 | 설명 |
|:---------|:-----|
| **컨**테이너 기술 발전 | Docker, Kubernetes 등 컨테이너 기반 배포 확산으로 빈번한 배포 필요 |
| **MSA** 확산 | 마이크로서비스 아키텍처 도입으로 다수 서비스의 독립적 배포 관리 필요 |
| **테**스트 중요성 대두 | 빈번한 변경에 따른 품질 보증을 위해 자동화된 테스트 필수 |

> 최근 MSA 및 컨테이너 기술의 발전에 따라 지속적인 통합 및 배포, 전달의 중요성으로 기업 내 CI/CD 파이프라인 구성이 활발하게 이루어짐

### 2. CI/CD 파이프라인 설명

{: .highlight }
> **CI/CD 파이프라인**: 지속적 통합, 지속적 배포, 자동화
> - (CI) `Code→Commit→Build→Test` (지속적 통합)
> - (CD) `Review→Staging→Production` (지속적 배포)
> - (구성요소) `버CI빌테코배모` (버전관리, CI서버, 빌드, 테스트, 코드품질, 배포, 모니터링)

#### 가. CI/CD 파이프라인 개념

```
┌─────────────────────────────────────────────────────────────────────────┐
│                         CI/CD 파이프라인                                │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  ┌────────────────────────────────────────────────────────────────┐    │
│  │                       CI Pipeline                              │    │
│  │  ┌──────┐   ┌──────┐   ┌──────┐   ┌────────┐   ┌───────────┐  │    │
│  │  │ Code │──▶│Commit│──▶│Build │──▶│  Unit  │──▶│Integration│  │    │
│  │  │      │   │      │   │      │   │ tests  │   │   tests   │  │    │

│  │  │      │        │         │        │            │            │    │
```

| 구분 | 정의 |
|:-----|:-----|
| **CI/CD** | 소프트웨어 개발 및 배포 프로세스를 지속적 통합과 지속적 배포를 통한 자동화 및 효율성을 극대화하기 위한 방법론 |
| **CI (Continuous Integration)** | 개발자들이 코드 변경사항을 공유 저장소에 자주 통합하고, 자동화된 빌드와 테스트를 수행하는 프로세스 |
| **CD (Continuous Delivery/Deployment)** | 통합된 코드를 스테이징, 프로덕션 환경에 자동으로 배포하는 프로세스 |

#### 나. CI/CD 구성요소 `버CI빌테코배모`

##### 그룹 1: 통합 및 빌드 도구 `버CI빌테`

| 구성요소 | 설명 | 도구 |
|:--------|:-----|:-----|
| **버**전관리 시스템 | 소스 코드를 저장하고 버전 관리를 수행하며 코드 변경 이력 추적 | Git, GitHub, GitLab, Bitbucket |
| **CI** 서버 | 개발자들이 코드 변경 사항을 커밋할 때마다 빌드, 테스트 실행 | Jenkins, GitLab CI, CircleCI, Travis CI |
| **빌**드 도구 | 소스 코드를 컴파일하고 패키징하며 코드의 종속성을 관리 | Maven, Gradle, Ant |
| **테**스트 프레임워크 | 테스트를 실행하여 코드의 기능적, 비기능적 요구 사항 검증 | JUnit, TestNG, Selenium, pytest |

##### 그룹 2: 품질 및 배포 도구 `코배모`

| 구성요소 | 설명 | 도구 |
|:--------|:-----|:-----|
| **코**드 품질 분석 도구 | 코드의 품질을 분석하고, 코드 스타일, 유지보수성 등 검토 | SonarQube, ESLint |
| **배**포 도구 | 애플리케이션을 스테이징 또는 프로덕션 환경에 배포 | Jenkins, GitLab CI, Spinnaker, Kubernetes |
| **모**니터링 및 로깅 도구 | 배포 애플리케이션 성능 및 상태 모니터링, 로그 수집 및 분석 | Prometheus, Grafana, ELK Stack |

> CI/CD 구성 시 보안 요소를 고려한 DevSecOps를 적용하여 파이프라인 구성을 수행함

### 3. CI/CD 파이프라인에서 DevSecOps 적용방안

{: .highlight }
> **DevSecOps 적용방안**: 초기 보안 통합, 자동화 테스트, 컨테이너 보안, 문화 구축
> - (개념) `Dev+Sec+Ops` - 개발 주기에 보안 측면 통합
> - (적용방안) `초자컨배피문` (초기보안, 자동화테스트, 컨테이너, 배포모니터링, 피드백, 문화)
> - (보안도구) `SAST/DAST` (정적/동적 보안 테스트)

| **핵심 원칙** | Shift-Left Security - 보안을 개발 초기 단계에서부터 통합 |
| **목적** | 보안 취약점을 조기에 발견하고 수정하여 비용 절감 및 품질 향상 |

##### 그룹 1: 보안 통합 단계 `초자컨`

| 단계 | 설명 | 도구/기법 |
|:-----|:-----|:---------|
| **초**기단계 보안 통합 | 개발 초기 단계부터 보안을 고려하여 코드 작성 및 검토를 진행, 이를 통해 보안 취약점을 조기에 발견하고 수정할 수 있음 | 보안 코드 리뷰, OWASP 가이드라인 |
| **자**동화된 보안 테스트 | 정적 애플리케이션 보안 테스트(SAST)와 동적 애플리케이션 보안 테스트(DAST)를 통합 | SAST(SonarQube, Checkmarx), DAST(OWASP ZAP, Burp Suite) |
| **컨**테이너 보안 | Docker 및 Kubernetes와 같은 컨테이너 환경에서 보안을 강화 | Trivy, Clair, Aqua Security |

##### 그룹 2: 운영 및 문화 단계 `배피문`

| 단계 | 설명 | 도구/기법 |
|:-----|:-----|:---------|
| **배**포 및 모니터링 자동화 | 배포 전 보안 검증 자동화 및 배포 후 실시간 모니터링 실시 | ArgoCD, Spinnaker, Prometheus, Grafana |
| **피**드백 및 개선 | 운영 환경에서 발생하는 보안 문제를 지속적으로 추적하고 개발팀과 보안팀 간의 실시간 소통을 통해 개선 | Slack, Jira, PagerDuty |
| **문**화 구축 | 팀 간 협업을 우선시하고, 보안이 모두의 책임이라는 인식을 공유하며 정기적인 교육과 워크숍을 진행 | 보안 교육, 워크숍, 챔피언 제도 |

### 4. DevSecOps 적용 시 보안 도구 비교

{: .highlight }
> **보안 테스트 도구**: SAST vs DAST
> - (SAST) 정적 분석 - 소스코드 레벨에서 취약점 탐지
> - (DAST) 동적 분석 - 실행 중인 애플리케이션 취약점 탐지
> - (IAST) 상호작용 분석 - SAST + DAST 결합

| 구분 | SAST (정적 분석) | DAST (동적 분석) | IAST (상호작용 분석) |
|:-----|:----------------|:-----------------|:-------------------|
| **분석 시점** | 개발 단계 (소스코드) | 실행 단계 (런타임) | 테스트 단계 (런타임) |
| **대상** | 소스코드, 바이너리 | 실행 중인 애플리케이션 | 실행 중인 애플리케이션 + 코드 |
| **장점** | 조기 발견, 수정 비용 절감 | 실제 환경 취약점 탐지 | 정확도 높음, 오탐 적음 |
| **단점** | 오탐(False Positive) 많음 | 개발 후반에 발견 | 에이전트 설치 필요 |
| **도구 예시** | SonarQube, Checkmarx, Fortify | OWASP ZAP, Burp Suite | Contrast Security, Synopsys |

---

### ⭐ 차별점 키워드

| 키워드 | 설명 |
|:------|:-----|
| **Shift-Left Security** | 보안을 개발 마지막이 아닌 초기 단계에서부터 통합 |
| **SAST/DAST 자동화** | CI/CD 파이프라인에 정적/동적 보안 테스트 자동 통합 |
| **컨테이너 보안** | Docker, Kubernetes 환경에서의 이미지 스캔 및 런타임 보안 |
| **보안 문화 구축** | "보안은 모두의 책임" - Security Champion 제도 |
