### 한 줄 정의

> **SP 인증**: SW 기업의 개발 프로세스 역량을 평가하여 1~3등급으로 인증하는 국가 품질보증 제도

### 쉬운 비유

| SP 인증 | GS 인증 |
|:-------|:-------|
| 요리사의 조리 실력을 평가 (요리 과정) | 완성된 요리의 맛을 평가 (결과물) |
| 프로세스(과정) 중심 | 제품(결과) 중심 |

### 왜 필요한가? (3줄 요약)

1. **SW 품질 일관성**: 개발 과정이 체계적이면 결과물도 안정적
2. **공공사업 입찰 가점**: SP 인증 기업에 입찰 시 인센티브 제공
3. **조직 역량 객관화**: CMMI 대비 국내 중소기업에 적합한 인증 체계

## 🎯 기술사 수준 설명

{: .highlight }
> **SP 품질인증**: 법적 근거 기반, 프로세스 성숙도 평가, 5개 영역 16개 항목 → SW 개발 역량 인증 제도
> - (1) `진흥법 23조` (개념/법적 근거)
> - (2) `프개지조프` (5개 영역)
> - (3) `1개2지프3조프` (등급별 심사영역)
> - ⭐ **차별점**: 프로세스 역량 중심 - GS인증(제품)과 달리 개발 과정의 성숙도 평가

### 1. 소프트웨어 프로세스 품질인증의 개념

{: .highlight }
> **개념**: SW 산업진흥법, 개발 역량 강화, 부실 방지
> - (법적 근거) `SW 산업진흥법 제23조`
> - (목적) `역품부` (역량 강화, 품질 향상, 부실 방지)
> - (인증체계) `과NIPA기` (과기부-NIPA-기업)

#### 가. 개념 정의

| 구분 | 내용 |
|:-----|:-----|
| **정의** | 개발 조직의 프로세스 역량을 정량적 기준으로 평가하여 SW 품질 확보 능력을 제도화한 국가 인증체계 |
| **법적 근거** | 「소프트웨어산업진흥법」 제23조, ISO/IEC 12207, 15504, CMMI 등 국제표준 준용 |

#### 나. 등장배경

```
🌐 과거: 중소 SW 기업의 개발 역량 편차 심각, 품질 불균형

💡 문제: "SW 사업 수행능력을 어떻게 객관적으로 증명할 것인가?"

✅ 해결: SW 산업진흥법 기반 SP 인증 제도 도입!
   → 프로세스 역량의 객관적 평가
   → 공공·민간 입찰 시 인센티브 제공
   → 중소기업 품질 경쟁력 강화
```

#### 다. 인증 관계도 (체계)

```
┌─────────────────┐
│    정책기관       │
│ (과학기술정보통신부) │◀──── 제도 정책 수립, 기준/지침 고시
└────────┬────────┘
         │ 정책지원 관리감독
         ▼
┌─────────────────┐          ┌─────────────────┐
│    인증기관       │          │    인증심의회     │
│     (NIPA)       │◀────────▶│                 │
└────────┬────────┘  인증심사결과 └─────────────────┘
         │                           ▲
         │ 인증심사                    │ 관리
         ▼                           │
┌─────────────────┐          ┌─────────────────┐
│    인증신청인     │          │    인증심사원     │
│   (SW 기업)      │◀─────────│                 │
└─────────────────┘  인증심사   └─────────────────┘
```

#### 라. 인증체계 구성요소 `과NIPA기`

| 구성요소 | 주요 역할 | 세부 내용 |
|:--------|:---------|:---------|
| **과**기부 (정책기관) | 제도 정책 수립 | 인증 기준, 지침 고시 등 |
| **NIPA** (인증기관) | 인증 업무 전반 주관 | 프로세스 인증심사, 인증심의회 운영, 심사원 관리, 시후관리 |
| **인**증심의회 | 인증 심사 결과 심의 | 인증 부여 여부 최종 결정 |
| **기**업 (신청인) | 인증 신청 | SW 프로세스 품질 인증 신청 및 표시 활용 |

### 2. 소프트웨어 프로세스 품질인증의 인증 기준 및 인증 등급

{: .highlight }
> **인증 기준**: 5개 영역, 16개 평가항목, 63개 세부항목
> - (5개 영역) `프개지조프` (프로젝트관리, 개발, 지원, 조직관리, 프로세스개선)
> - (등급) `1개2지프3조프`
> - (등급 의미) `프조조` (프로젝트→조직차원→조직개선)

#### 가. 인증 기준 (5개 영역)

```
                    ┌──────────────────────────────────────┐
                    │      영역별 평가항목 (총 16개)         │
                    │           대상: 1/2/3등급            │
                    └──────────────────────────────────────┘
                                       │
    ┌──────────┬──────────┬───────────┼───────────┬──────────────┐
    ▼          ▼          ▼           ▼           ▼              ▼
┌────────┐ ┌────────┐ ┌────────┐ ┌────────┐ ┌────────────┐
│프로젝트 │ │  개발   │ │  지원   │ │ 조직   │ │ 프로세스   │
│ 관리   │ │  영역   │ │  영역   │ │ 관리   │ │   개선     │
└────────┘ └────────┘ └────────┘ └────────┘ └────────────┘
 계획,통제   요구,분석   품보,형상   조직관리   조직성과관리
 협력업체   설계,구현   측정,분석   구성원교육  문제해결
             테스트                          프로세스개선
```

| 영역 | 평가항목 | 대상 등급 |
|:-----|:---------|:--------:|
| **프**로젝트 관리영역 | 프로젝트 계획, 프로젝트 통제, 협력업체 관리 | 2, 3등급 |
| **개**발영역 | 고객 요구사항 관리, 분석, 설계, 구현, 테스트 | 1, 2, 3등급 |
| **지**원영역 | 품질보증, 형상관리, 측정 및 분석 | 2, 3등급 |
| **조**직관리영역 | 조직 프로세스 관리, 구성원 교육 | 3등급 |
| **프**로세스개선영역 | 조직성과 관리, 문제해결, 프로세스 개선 | 3등급 |

> 총 **5개 영역**, **16개 영역별 평가항목**, **63개 세부평가항목**으로 품질인증 기준 구성

#### 나. 인증 등급 `1개2지프3조프`

```
┌─────────────────────────────────────────────────────────────────────┐
│                    SP 품질인증 등급별 심사영역                         │
└─────────────────────────────────────────────────────────────────────┘

┌─────────┐      ┌─────────────┐      ┌─────────────────────┐
│  1등급   │ ──▶ │    2등급     │ ──▶ │       3등급          │
└─────────┘      └─────────────┘      └─────────────────────┘
     │                  │                      │
     ▼                  ▼                      ▼
┌─────────┐      ┌─────────────┐      ┌─────────────────────┐
│   개발   │      │    개발      │      │    프로세스 관리     │
│  심사영역 │      │    지원      │      │     조직관리        │
│         │      │ 프로젝트 관리 │      │      개발           │
│         │      │   심사영역    │      │      지원           │
│         │      │             │      │   프로젝트 관리      │
│         │      │             │      │     심사영역        │
└─────────┘      └─────────────┘      └─────────────────────┘
```

| 등급 | 심사영역 | 역량 수준 |
|:---:|:---------|:---------|
| **1등급** | 개발 | 프로젝트 성공 여부와 관계없이 특정 프로젝트를 수행할 수 있는 수준, 프로세스 역량 개선이 필요한 수준 |
| **2등급** | 개발, 지원, 프로젝트 관리 | 프로젝트 차원의 프로세스가 수립되고, 이를 기반으로 프로젝트를 통제하여 성공적으로 수행할 수 있는 역량 수준 |
| **3등급** | 프로세스 관리, 조직관리, 개발, 지원, 프로젝트 관리 | 조직 차원의 프로세스 체계 정의, 정량적 데이터 관리, 문제의 근본 원인 해결, 지속적 프로세스 개선 가능 역량 |

### 3. SP 인증과 GS 인증 비교

{: .highlight }
> **SP vs GS 비교**: 프로세스 vs 제품, 개발역량 vs 품질적합성
> - (SP) `과정중심` - 개발 프로세스 및 관리체계 평가
> - (GS) `결과중심` - 완성된 제품의 품질 평가

| 구분 | SP 인증 | GS 인증 |
|:-----|:--------|:--------|
| **법적 근거** | 소프트웨어 진흥법 제21조 | 소프트웨어 진흥법 제20조 |
| **평가 대상** | 개발조직의 SW 개발 프로세스 및 관리체계 | 완성된 SW 제품 자체의 기능·성능·호환성 |
| **평가 기준** | ISO/IEC 12207, 15504, CMMI 기반 프로세스 성숙도 및 일관성 | ISO/IEC 25023, 25041 기반 제품 품질특성(기능성·신뢰성·사용성·효율성) |
| **목적** | 조직의 개발 역량 강화 및 품질 관리 체계 확립 | 상용 SW 제품의 품질 적합성 보증 및 시장 신뢰 확보 |
| **운영 기관** | 한국정보통신기술협회(TTA), 한국소프트웨어산업협회(KOSA) 공동 운영 | 한국정보통신기술협회(KTL) 외 4곳 |

### ⭐ 차별점 키워드

| 키워드 | 설명 |
|:------|:-----|
| **프로세스 역량 중심** | 결과물(제품)이 아닌 개발 과정의 성숙도를 평가 |
| **국가 인증체계** | 과기부-NIPA 기반 공식 인증, 공공사업 입찰 인센티브 |
| **등급별 심사영역 확장** | 1등급(개발) → 2등급(+지원,관리) → 3등급(+조직,개선) |
| **CMMI 국내화** | 국제 표준을 국내 중소기업에 맞게 간소화 |
