### 🔑 핵심 키워드 3개

| 키워드 | 설명 |
|:------|:-----|
| **협업** | 개발팀과 운영팀이 벽을 허물고 한 팀처럼 일하는 것 |
| **자동화** | 코드 테스트부터 배포까지 기계가 알아서 척척 |
| **CI/CD** | 코드 변경→테스트→배포가 쉬지 않고 계속 돌아가는 파이프라인 |

---

### 📖 등장배경

🌐 **과거**: 개발팀은 코드만 만들어서 운영팀에 던지고, 운영팀은 "왜 이렇게 만들었어? 서버에서 안 돌아가잖아!" 하면서 싸웠어요.

💡 **문제**: "개발은 끝났는데 배포는 왜 이렇게 오래 걸려? 새 기능 하나 출시하는데 몇 달씩?"

✅ **해결**: DevOps로 개발과 운영이 한 팀이 되자!
   → 개발자와 운영자가 함께 협업
   → CI/CD 파이프라인으로 자동화
   → 빠른 피드백과 신속한 배포

---

### 📝 개념 정의

> **DevOps**: 개발자(Developer)와 운영자(Operations)가 협업하여 빠르게 서비스를 만들고 배포하는 문화이자 방법론

### 쉬운 비유

| 기존 방식 | DevOps 방식 |
|:-----|:-----|
| 요리사가 음식 만들어서 서빙 직원에게 던짐 (서로 탓함) | 요리사와 서빙 직원이 함께 주문받고, 요리하고, 손님 피드백까지 같이 처리 |

---

### 🏗️ 기술요소 (2그룹 × 4가지)

#### 그룹 1: 장점 `안신테협`

| 항목 | 설명 |
|:-----|:-----|
| **안**정성 향상 | 릴리즈와 배포 분리로 안정된 버전업 |
| **신**속한 배포 | CI/CD로 Time to Market 단축 |
| **테**스트 자동화 | SonarQube, JUnit 등으로 품질 UP |
| **협**업 강화 | 개발/운영 간 Silo(벽) 극복 |

#### 그룹 2: 단점 `난복역보`

| 항목 | 설명 |
|:-----|:-----|
| **난**이도 | 문화, 조직 변화 저항 |
| **복**잡도 | 도구, 파이프라인 관리 부담 |
| **역**량 | 다기능 인력 필요 |
| **보**안 리스크 | 자동화 미성숙 시 설정 오류 확산 |

---

### ⭐ 차별점 키워드

**"개발과 운영의 경계를 없앤 것"** - 기존엔 만드는 사람/돌리는 사람이 따로였는데, DevOps는 둘을 하나로 합침

---

### 📋 6하원칙 요약

| 구분 | 내용 |
|:-----|:-----|
| **누가(Who)** | 개발자 + 운영자 (하나의 팀) |
| **언제(When)** | SW 개발~운영 전 과정에서 |
| **어디서(Where)** | 클라우드, 온프레미스 등 인프라 환경 |
| **무엇을(What)** | CI/CD 파이프라인을 통한 자동화된 빌드/테스트/배포 |
| **왜(Why)** | 빠른 출시, 안정적 운영, 고객 피드백 반영 |
| **어떻게(How)** | 협업 문화 + 자동화 도구(Jenkins, Docker, K8s 등) |

</details>

## 🎯 기술사 수준 설명

{: .highlight }
> **DevOps**: SW 개발과 운영 단절 문제 → 협업, 자동화, CI/CD → 개발·운영 통합 패러다임
> - (장점) `안신테협`
> - (단점) `난복역보`
> - ⭐ **차별점**: 개발과 운영의 Silo를 허물어 지속적 전달(CD)을 실현하는 문화·기술 통합 접근

---

### 🔑 핵심 키워드 3개

| 키워드 | 설명 |
|:------|:-----|
| **협업(Collaboration)** | 개발/운영 조직 간 사일로(Silo) 극복을 위한 소통과 협력 |
| **자동화(Automation)** | CI/CD 파이프라인을 통한 빌드, 테스트, 배포 자동화 |
| **통합(Integration)** | 개발과 운영 프로세스의 연속적 통합 |

---

### 📖 등장배경

🌐 **과거**: 개발팀과 운영팀이 분리된 조직(Silo) 형태로 운영

💡 **문제**: "개발 완료 후 운영 이관 시 충돌, 배포 지연, 장애 발생 시 책임 회피"

✅ **해결**: DevOps로 개발과 운영의 협업 문화와 자동화 도구 도입!
   → 개발·운영 조직 간 소통 및 협업 강화
   → CI/CD 파이프라인을 통한 배포 자동화
   → 피드백 루프를 통한 지속적 개선

---

### 📝 개념 정의

> **DevOps**: 소프트웨어 개발자와 운영자 간의 소통, 협업, 통합, 자동화를 통해 CI/CD를 실현하는 개발 패러다임

| 구분 | 정의 (22자 이내) |
|:-----|:----------------|
| **CI (Continuous Integration)** | 코드 변경의 지속적 통합 및 검증 |
| **CD (Continuous Delivery/Deployment)** | 지속적 전달 및 배포 자동화 |

---

### 🏗️ 구성요소

#### 그룹 1: DevOps 장점 `안신테협`

| 항목 | 설명 | 예시 |
|:-----|:-----|:-----|
| **안**정성 향상 | 릴리즈와 배포 분리 통한 안정된 버전업 가능 | Blue-Green, Canary 배포 |
| **신**속한 배포 | CI/CD 파이프라인 통한 적시 출시(Time to Market 단축) | Jenkins Pipeline, GitLab CI |
| **테**스트 자동화 | 테스트 도구 도입으로 품질 향상 | SonarQube, TDD, JUnit |
| **협**업 강화 | 개발/운영 간 Silo 극복, 피드백 루프 형성 | ChatOps, 공동 On-Call |

#### 그룹 2: DevOps 단점 `난복역보`

| 항목 | 설명 | 예시 |
|:-----|:-----|:-----|
| **난**이도 | 문화, 조직 변화 저항 | 기존 Silo 조직의 변화 거부 |
| **복**잡도 | 도구, 파이프라인 관리 부담 | Git, K8s, Jenkins, Docker 학습 |
| **역**량 | 다기능 인력 필요 | 멀티스킬 엔지니어 희소성 |
| **보**안 리스크 | 자동화 미성숙 시 설정 오류 확산 | CI/CD 파이프라인 보안 취약점 |

### ⭐ 차별점 키워드 (상세)

| 구분 | 차별점 |
|:-----|:-------|
| **vs 전통적 개발** | Silo 조직 → 협업 조직, 수동 배포 → 자동화 배포 |
| **vs Agile** | Agile은 개발 방법론, DevOps는 개발+운영 통합 문화 |
| **핵심 차별점** | 개발과 운영의 경계를 없애고 피드백 루프를 통한 지속적 개선 |

---


### 🔧 DevOps Best Practice

| Best Practice | 설명 |
|:---|:---|
| **SRE** | 에러 버킷(Error Budget) 관리를 통한 구글의 DevOps 방법론 |
| **IaC** | 인프라 설정을 코드화하여 인프라 관리 자동화 (Terraform, Ansible) |
| **CI/CD 자동화** | 코드→테스트→배포→릴리즈 자동화로 속도와 안전성 증가 |
